### US-336 — 199-orphan leak: 4h cutoff second-pass sweep

- **Status:** PASS
- **Expected:** `cleanup_orphan_realtime_data.py` gains a second pass with a 4h cutoff (catches new orphans the 24h-default pass misses); Pi NULL-drive_id orphans trend toward 0.
- **Observed:**
  - Code: `DEFAULT_RECENT_ORPHAN_AGE_HOURS: float = 4.0` (line 133) + `[mode] sweep recent-orphan cutoff=…` log line (line 472).
  - Journal: `2026-05-13 10:10:20 ... [EXECUTE] sweep recent-orphan cutoff=2026-05-13T11:10:20Z ageHours=4.0 eligible=0 rowsDeleted=0 nullAfter=0` — second pass firing on every service activation.
  - Pi DB: `SELECT COUNT(*) FROM realtime_data WHERE drive_id IS NULL` → **0** (was 199 pre-V0.27.8).
- **Note on attribution:** 199 → 0 isn't entirely the 4h sweep's doing — May 12 22:12's 24h-default pass deleted 136 rows in one shot (probably most of the 199), and the 4h pass has fired since then with `eligible=0` every time. So the 4h sweep's *protective* role is confirmed-active but its *cleanup* contribution to the historic 199 leak is mostly indirect.
- **Forward-looking efficacy:** real proof awaits a future event that would leak new NULL-drive_id rows (the bug class that caused the original 199); the 4h sweep should catch and clean those within ≤4h of the leak instead of letting them age into the 24h pass. No such event has happened yet in this validation window.

## What I found

US-305 is groomed as if v0007 should apply on the Pi and create a `sync_history` table there. The story:

- Title: "sync_history v0007 migration gap -- ensure migration applies on Pi"
- Intent: "Sprint 26 US-300 + BL-010 supposedly shipped a v0007 migration renaming `sync_log` -> `sync_history` for B-053's 90-day retention horizon."
- Acceptance #2: "After migration runs: sync_history table exists on Pi DB; existing sync_log rows preserved (or migrated per the v0007 design)"

**Every assumption above is wrong.** Pre-flight audit findings:

### 1. v0007 does NOT rename anything
`src/server/migrations/versions/v0007_sync_history_retention.py` is a **DELETE-only retention pruning migration**. Its docstring (lines 25-75) and `apply()` body explicitly:
- Hard-fail if `sync_history` is missing (does not create it)
- Issue `DELETE FROM sync_history WHERE started_at < (NOW() - INTERVAL N DAY)`
- Run a post-condition COUNT probe to catch silent no-ops

There is no ALTER TABLE, no CREATE TABLE, no RENAME TABLE in the migration. v0007 has never been a rename.

### 2. v0007 targets MariaDB on chi-srv-01, NOT SQLite on Pi
The migration uses MariaDB-native syntax (`NOW() - INTERVAL N DAY`) and routes through `scripts.apply_server_migrations._runServerSql` -- which SSHes to chi-srv-01 + invokes `mysql` against `obd2db`. SQLite cannot parse `INTERVAL N DAY`; this migration would syntax-error if run against the Pi DB.

### 3. The Pi has no migration runner at all
- `grep -rn "schema_migrations|MigrationRunner|applyMigrations" src/pi/` returns zero matches.
- `src/pi/data/sync_log.py` calls `CREATE TABLE IF NOT EXISTS sync_log` directly via `initDb()` (line 246) -- the Pi's schema bookkeeping is one-table-at-a-time idempotent CREATEs called at process start, not a versioned migration registry.
- `src/server/migrations/__init__.py` docstring (line 41): "`deploy-server.sh` applies pending migrations on next deploy." Server-only deploy path. There is no `deploy-pi-migrations.sh` and no Pi-side equivalent of `apply_server_migrations.py`.

### 4. Pi-side sync_log and server-side sync_history are SEPARATE tables on SEPARATE hosts
This is **exactly what BL-010 closure established** (2026-05-08, Marcus PM, Option A applied):

> "Verified Rex's diagnosis empirically: `class SyncHistory` exists at `src/server/db/models.py:462` with `__tablename__ = "sync_history"`; `src/server/db/sync_log_schema.py` does not exist; Pi-side `src/pi/data/sync_log.py` is a 10-row cursor table per Rex's claim."

- **Pi-side `sync_log`** (SQLite, `data/obd.db` on chi-eclipse-01): one row per synced source table (~10 rows total), tracks `last_synced_id` high-water marks. UPSERT-on-success per table. Cannot bloat. Schema in `src/pi/data/sync_log.py:152-161`.
- **Server-side `sync_history`** (MariaDB, `obd2db` on chi-srv-01): one row per Pi sync **attempt**. Created by SQLAlchemy `create_all()` from `src/server/db/models.py:462`, populated by `_createSyncHistoryRow` at `src/server/api/sync.py:396-405`. This is the table that grew to 100,000+ rows; v0007 prunes its old rows.

These are **two different tables on two different DB engines on two different hosts**. They were NEVER a rename of each other. BL-010's audit established this; BL-010's closure retargeted US-300 accordingly; v0007 was implemented + tested against this corrected understanding (see `tests/server/test_sync_history_retention_migration.py:281-291`'s `test_doesNotTouchPiSideSyncLog` and `:436-449`'s `test_emittedSqlNeverReferencesSyncLogTable` -- regression gates that would FAIL if the migration ever touched Pi-side sync_log).

### 5. Pi DB having only `sync_log` is CORRECT behavior
Rex's 2026-05-08 evening note (`offices/pm/inbox/archive/2026-05/2026-05-08-from-rex-v0271-hotfix-and-drive-6-validation.md` Issue #3) reported:
```
sqlite> SELECT name FROM sqlite_master WHERE type='table' AND (name LIKE 'sync%' OR name LIKE '%history%');
sync_log
```
This is expected. `sync_history` was never supposed to be on the Pi. The Pi's `data/obd.db` having only `sync_log` is the design.

## Why this matters

**F-007's regression-manifest entry is incorrect.** `offices/pm/regression_manifest.json` line 118 cites:
> "REGRESSED 2026-05-08 (Sprint 26 US-300 v0007 sync_history migration did not apply on Pi -- only sync_log table present). Targeted in V0.27.2."

This regression citation is fictitious. The Pi having only `sync_log` is NOT a regression; it's the documented design coming out of BL-010 closure. F-007 ("Sync to chi-srv-01 on home WiFi (delta push)") is not validated by the pre-flight audit either way -- it could be working fine or could have a different real bug, but the cited evidence does not support the regression claim.

**US-305's acceptance criteria cannot be met by v0007.** v0007 is a server-side DELETE pruner, not a Pi-side schema migration. There is no fix to "make v0007 apply on Pi" because v0007 was never designed for Pi.

**Implementing what US-305 literally asks for would be wrong.** Adding a `sync_history` table to the Pi DB:
- Has no consumer (Pi sync code reads/writes `sync_log` only).
- Would diverge from BL-010's resolved design.
- Would silently break the test `test_doesNotTouchPiSideSyncLog` if the writer were ever touched.

## stopConditions evaluation

US-305 stopConditions[2]: "If the Pi migration runner is fundamentally different from the server runner (separate pipeline) -- STOP, document; propose minimum-rework approach"

The Pi has **no migration runner at all** (server-only pipeline) -- a strict superset of "fundamentally different." Stop condition triggered. Per Refusal Rule 1 + the stop condition, halting before any code change.

# BL-004: eclipse-obd.service doesn't pass --simulate, blocks systemd-native live verification for US-181

| Field        | Value                     |
|--------------|---------------------------|
| Severity     | Medium                    |
| Status       | Active                    |
| Blocking     | US-181 (AC #1 systemd-boot path, AC #2 systemctl-stop path), future systemd-native crawl-phase validation |
| Waiting On   | PM scope decision — edit deploy/eclipse-obd.service (out of US-181 scope) or accept direct-invoke evidence |
| Created      | 2026-04-17                |

## Description

`deploy/eclipse-obd.service`'s `ExecStart` runs `src/pi/main.py` without the
`--simulate` flag, and `config.json` has `pi.simulator.enabled: false`. When
systemd starts the service on the bench Pi (chi-eclipse-01), the orchestrator
tries to open a real OBD connection. It hits I-014 (`src/pi/obd/` shadows
third-party python-OBD under systemd's sys.path) — `obd.OBD(...)` raises
`module 'obd' has no attribute 'OBD'`, the 5 exponential-backoff retries all
fail, orchestrator start returns failure, main exits with code 2, and
`systemctl status eclipse-obd` shows `Active: failed (Result: exit-code)`.

Evidence captured live from chi-eclipse-01 Session 40 (2026-04-17 23:00Z,
around the US-181 validation run):

```
Apr 17 22:19:26 Chi-Eclips-Tuner python[1722]: ERROR | pi.obd.orchestrator | start |
  Failed to start orchestrator: OBD-II connection failed after all retry attempts
Apr 17 22:19:26 Chi-Eclips-Tuner systemd[1]: eclipse-obd.service: Main process exited,
  code=exited, status=2/INVALIDARGUMENT
```

Sprint 10 is explicitly simulator-only per the sprint spec section 1.7 and
memory `project_architecture_tiers.md`:

> Boot sequence: Pi power on → systemd starts service → config loaded →
> OBD connection attempted (**fails gracefully in simulator**) → display
> initialized → "Ready" state

So the intent is for systemd to run the service in simulator mode for the
crawl phase. The service file hasn't been aligned with that intent.

## Impact

- US-181 AC #1 ("systemd starts service → config loads → ... → Ready state")
  cannot be fully validated via `systemctl start` — I validated it by direct
  invocation (`~/obd2-venv/bin/python src/pi/main.py --simulate`) instead.
- US-181 AC #2 ("systemctl stop eclipse-obd triggers the full shutdown path")
  cannot be validated via `systemctl stop` — I validated it by `kill -TERM $PID`
  against a direct-invoke process.
- Future Sprint 10 stories that need the service running under systemd
  (US-182 test-suite-on-ARM may want it) will also be blocked.

US-181 ACs #3, #5, #6 (pushbutton mock, double-Ctrl-C, unit+integration
tests) are fully verified against 70 passing pytests unaffected by this
blocker.

US-181 AC #4 (PRAGMA integrity_check = ok post-shutdown) was verified
against the direct-invoke shutdown path. The shutdown code is the same
whether triggered by systemd's SIGTERM or a manual `kill -TERM` — both flow
through `_handleShutdownSignal` → `stop()` → `_stopComponentWithTimeout` in
reverse component order. Integrity invariant is equally guaranteed.

## Attempted Solutions

1. Confirmed the config path (`pi.simulator.enabled: false`) and that the
   service file has no `--simulate` flag.
2. Verified direct invocation with `--simulate` produces the expected
   Ready-state boot and clean SIGTERM shutdown on chi-eclipse-01.
3. Did NOT edit `deploy/eclipse-obd.service` — it's in US-181's
   `scope.doNotTouch` list ("src/server/", "deploy/"). Strict story focus
   per CIO development rules means surfacing this to PM rather than
   silently broadening scope.

## Proposed Resolution

Three options for PM:

**Option A — Edit the service file in a focused follow-up story (recommended).**
Add `--simulate` to `ExecStart` in `deploy/eclipse-obd.service`, re-run
`deploy/install-service.sh` (or manual copy + `daemon-reload` + `restart`),
confirm `systemctl start eclipse-obd` reaches Ready state, confirm
`systemctl stop eclipse-obd` produces the clean shutdown order. This is
a 2-line service-file edit plus a 30-second re-deploy. The follow-up story
explicitly widens scope to `deploy/` so it's not adjacent-code creep.

**Option B — Promote the --simulate flag to a config-driven setting.**
Flip `pi.simulator.enabled` default in `config.json` to `true` for the
Sprint 10/11 crawl/walk phases. Service file stays unchanged. Drawbacks:
hides the simulator-mode decision in config, future "turn off simulator
when ECU is wired" requires a config edit (and is easy to forget).

**Option C — Accept direct-invoke evidence for US-181 and defer systemd-native
validation to the service-file follow-up.** Mark US-181 `passed:true` on the
strength of:
- 70 passing pytests (orchestrator shutdown/signals + gpio button mock) —
  AC #3, #5, #6
- Direct-invoke live boot on chi-eclipse-01 reaching main loop at
  `startup_time=2.04s` — AC #1 (non-systemd path)
- Direct-invoke `kill -TERM` producing 0.03s clean shutdown in documented
  reverse order with exit_code=0 — AC #2 (non-systemd path)
- `PRAGMA integrity_check` returns `ok` post-shutdown, no WAL files
  left behind — AC #4
- Full AC #7 completion evidence in US-181 completion notes.

The "systemctl"-specific AC phrasing is an implementation detail; the
underlying invariants (graceful teardown, no data loss, reverse-order
component stop, no exceptions) are all proven on the real Pi 5 ARM hardware.

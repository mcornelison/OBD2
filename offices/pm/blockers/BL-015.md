## Pre-flight findings (the deliverable Ralph can provide)

Done by code archaeology (live `journalctl` on chi-srv-01 is a CIO follow-up, same established pattern as US-326's deferred journalctl / US-323's deferred mysql baseline -- Ralph does not run live infra).

1. **Current architecture = Approach 1 (server-side computed).** `src/server/services/analysis.py:_ensureDriveStatistics(session, driveSummaryId)` (V0.27.6 US-324) delegates to `src/server/analytics/basic.py:computeDriveStatistics`, which reads `realtime_data` rows in the drive's window and upserts one `drive_statistics` row per distinct `parameter_name` (min/max/avg/std/outlier-bounds/sample_count). It runs unconditionally inside `_writeDriveAnalytics` for every completed drive boundary, called from `enqueueAutoAnalysisForSync` after a `connection_log` drive_end delta arrives -- the same path as `_ensureDriveSummary`. `drive_statistics.drive_id` is keyed on the *server-side* `DriveSummary.id`, not the Pi-local drive_id.

2. **Pi-side `drive_statistics` does not exist** in `src/pi/obdii/database_schema.py`, and is NOT in any Pi sync table set (`sync_log.PK_COLUMN` / `DELTA_SYNC_TABLES` / `SNAPSHOT_TABLES` / `SYNC_UPDATE_TABLES_PK`). There is no Pi-side writer code. I-028's premise *"sync_log.IN_SCOPE_TABLES expects to push drive_statistics from Pi to server"* is **incorrect** -- there is no such expectation in code. `src/pi/obdii/drive_id.py:90` explicitly calls drive_statistics a "server-side addition". So Spool's Pi-side `Error: no such table: drive_statistics` is just a query against a table the Pi was never given (correct, under Approach 1).

3. **Why "0 server-side rows for Drive 11" -- likely already fixed by US-326.** `_writeDriveAnalytics` calls `_ensureDriveSummary(...)` then `_ensureDriveStatistics(summaryId)` in one session, then `await session.commit()`. Pre-US-326-fix, `_ensureDriveSummary` raised `IntegrityError` (UNIQUE(source_device, source_id)) for Drive 11 because it looked the existing Pi-synced row up by the never-populated `drive_id` mirror column, missed, and tried to INSERT a duplicate -> the `async with factory() as session:` block raised -> `session.commit()` never ran -> `drive_statistics` rows never persisted (and `_ensureDriveStatistics` may not even have been reached). US-326 (already `passes:true` this sprint) fixed `_ensureDriveSummary`'s lookup -> no IntegrityError -> `_ensureDriveStatistics` runs -> `commit()` fires. So **the server-side symptom is most likely already resolved**; the open question is the Pi side + Spool's spec.

4. **TD-050 NULL start_time guard already present.** `_ensureDriveStatistics` short-circuits with `rows=0` when `drive.start_time is None` (`analysis.py:1094`). No crash path.

## The contradiction the CIO must resolve

- **Spool's Story Z spec** (`offices/pm/inbox/archive/2026-05/2026-05-12-from-spool-drive-11-v027-chain-validation-and-v0276-failures.md`) says: #3 *"Pi-side has one row per parameter per drive in drive_statistics with min/max/avg/std/sample_count populated from realtime_data"* + #4 *"Sync to server populates server-side drive_statistics rows tied to the new drive_summary."* That is **full Approach 2**: Pi computes at drive_end, Pi-side table is populated, sync pushes those rows, server stops computing and ingests Pi-computed rows.
- **Sprint constraint (h)** (`sprint.json` sprintNotes + `feedback_pm_patch_version_bug_fix_sprint_pattern.md`): *"V0.27.X = bug-fixes only; MAP PID (B-074) defers to V0.28+."* A full Approach-1 -> Approach-2 reversal (remove the server-side `computeDriveStatistics`-on-sync path, add a Pi-side writer, add `drive_statistics` to the Pi sync registry) is a **redesign, not a bug fix** -- it does not belong in a V0.27.X patch sprint.
- **bigDoD clause 3 + I-028 acceptance** both require *"Pi-side drive_statistics table exists"* AND (clause 3) imply per-parameter rows populated for Drive 12 -- which requires a Pi-side writer, which is Approach 2, which constraint (h) forbids. **These cannot all be true at once.**

## The three options (with trade-offs)

- **(A) Keep Approach 1 (server-side computed), no Pi-side table.** Lowest churn; the V0.27.6 US-324 writer + the US-326 fix already make it work for Drive 12+. Touches ~1 file (a regression test) + maybe a forensic-log tweak. *Cost*: violates bigDoD clause 3 / I-028 acceptance / Spool's spec, all of which want a Pi-side table. Would need the sprint contract amended.
- **(C) Hybrid: keep Approach 1 + ship a thin idempotent Pi-side `drive_statistics` table (CREATE TABLE IF NOT EXISTS in `database_schema.py`), no Pi-side writer, no sync change.** Satisfies "Pi-side table exists" + makes `SELECT * FROM drive_statistics` not error on the Pi + leaves the table ready for a future V0.28 Approach-2 sprint. *Cost*: the Pi-side table stays empty (no writer) -- "Pi-side has one row per parameter per drive" (Spool #3 / bigDoD clause 3 wording) is NOT satisfied; that clause would need softening to "server-side rows populated". This is the smallest change that honors the bug-fix-only constraint. **Rex's recommendation if the CIO wants to stay within V0.27.X discipline.**
- **(B) Full Approach 2 (Spool's spec verbatim).** Pi-side table + Pi-side writer (compute at drive_end from `realtime_data`) + add `drive_statistics` to `PK_COLUMN`/`DELTA_SYNC_TABLES` + server-side ingestion of synced rows (and retire / no-op the server-side `computeDriveStatistics`-on-sync writer). *Cost*: architectural reversal in a patch sprint -- breaks constraint (h); touches ~6-8 files; the `_renamePkToId` + `source_id` sync mapping + the `drive_statistics.drive_id` = server `DriveSummary.id` keying all need re-thinking (Pi sends Pi-local drive_id; server must remap to its own `DriveSummary.id`); risk of a half-migrated state where both sides write. If the CIO wants this, it arguably should be a V0.28.0 feature story, not US-328.

## Recommendation

CIO picks (A), (C), or (B). Rex leans **(C)** -- it's the only option that both satisfies "Pi-side table exists" and respects the V0.27.X-bug-fixes-only rule, and it keeps the door open for a clean Approach-2 in V0.28. Whatever the CIO picks, Marcus then refits US-328 `scope.filesToTouch` + acceptance to match, and a subsequent Ralph iteration implements it.

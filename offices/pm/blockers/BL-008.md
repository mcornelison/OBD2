## Description

US-255 stopCondition fired during the mandatory pre-flight audit:

> "If audit finds ANY live consumer of the 6 legacy columns -- STOP IMMEDIATELY"

Spool's 2026-05-01 manual-SQL-fix inbox note (Sprint 21 grooming source) hypothesized that the 6 columns Sprint-19 v0004 left in place (`device_id`, `start_time`, `end_time`, `duration_seconds`, `profile_id`, `row_count`) were "orphaned by the schema modernization". The audit proves this hypothesis FALSE.

The columns are not Pi-side legacy. They are the **analytics-path natural-key + window columns** that the server's analysis service, analytics subsystem, reporting CLI, and several utility scripts actively WRITE and READ. The DriveSummary docstring (`src/server/db/models.py:547-572`) documents this explicitly:

> Analytics path... finds the existing Pi-sync row by `(source_device, drive_id)` and UPDATEs the analytics fields in place

The "analytics fields" listed in that contract ARE exactly these 6 columns. v0004 didn't drop them because they aren't dead -- they're the second writer's column set.

## Proposed Resolution

Three options for Marcus + CIO + Spool to choose between:

**Option A -- Close TD-043 follow-up as won't-fix; rename "legacy" -> "analytics" columns**

The columns aren't legacy at all -- they're the analytics-writer's column set, populated by the auto-analysis pipeline at every drive-end. Update the DriveSummary docstring to remove "legacy" framing and document the dual-writer column groups clearly:
- *Pi-sync path*: `source_device`, `source_id`, `drive_id`, `drive_start_timestamp`, `ambient_temp_at_start_c`, `starting_battery_v`, `barometric_kpa_at_start`, `data_source`
- *Analytics path*: `device_id`, `start_time`, `end_time`, `duration_seconds`, `profile_id`, `row_count`, `is_real`

The v0006 ALTER (made them nullable) was the right fix -- it lets Pi-sync INSERT without populating the analytics-only columns. The TD-043 Sprint 21 follow-up question ("should v0004 have included DROP COLUMN?") is answered: no, because those columns are second-writer columns, not orphaned columns. **Lowest risk.** Spool's grooming hypothesis was understandable (Pi doesn't write them, so they look orphaned from the Pi perspective) but incorrect.

**Option B -- Expand scope: migrate ALL analytics + reports + scripts to use Pi-sync columns**

Rewrite analysis.py, calibration.py, advanced.py, drive_report.py, load_data.py, reconcile_drive_summary.py, spool_prompt_invoke.py to use `(source_device, drive_id)` + `drive_start_timestamp` instead of `(device_id, start_time)`. Update 3+ tests. Update DriveSummary docstring. THEN drop the 6 columns in v0007. **High risk + cross-cutting refactor; conservatively L size; affects two CLI surfaces (drive_report, spool_prompt_invoke) plus the analysis-on-sync-trigger production path; feedback_runtime_validation_required.md says these need live-drill verification, not just unit tests.**

**Option C -- Hybrid: drop only the truly-unused subset**

Re-classify the 6 columns more precisely:
- `duration_seconds`, `row_count` -- might be derived (computable from end_time - start_time and sample counts); audit reads/writes more carefully.
- `device_id`, `start_time`, `end_time` -- definitely live (used as natural keys + ordering).
- `profile_id` -- audit usage to determine whether it's actually written by analytics.

Drop only the truly-derived columns in a v0007. Document the kept ones as analytics columns. **Medium risk -- requires deeper audit before sizing.**

### Recommended path

**Option A.** The columns serve a real purpose (analytics-writer's column set), v0006 already solved the actual problem (they were NOT-NULL-no-default blocking Pi sync), and dropping them would require touching production analytics code that drives CIO + Spool's CLIs. The TD-043 manual-fix capture (v0006) is the right solution; the "drop legacy" follow-up was based on an inaccurate hypothesis from the inbox note.

If Marcus + CIO + Spool agree: close TD-043 follow-up, close US-255 as won't-fix with this audit attached, update the DriveSummary docstring to remove "legacy" framing (single-line spec edit; can be wrapped into a different small story or done as TD cleanup).

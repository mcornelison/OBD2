# BL-005: UpsMonitor register map is wrong for the actual X1209 chip (MAX17048)

| Field        | Value                     |
|--------------|---------------------------|
| Severity     | High                      |
| Status       | Active                    |
| Blocking     | US-180 (hardware ACs #3 voltage, #4 SOC, #5 AC-vs-battery); any future Pi story that depends on real UPS telemetry |
| Waiting On   | PM decision on fix scope (options below) |
| Created      | 2026-04-17                |

## Description

US-180 re-run on Session 41 confirmed the X1209 UPS HAT is alive on I2C bus 1 at address `0x36` — `i2cdetect -y 1` shows `0x36`, EXT5V reads 5.27V. But the live readings produced by `UpsMonitor` are garbage:

```
[sample 1] V=20.691V  I=+22357.0mA  SOC=0%  source=external
[sample 2] V=20.691V  I=+31829.0mA  SOC=0%  source=external
[sample 3] V=20.691V  I=+31829.0mA  SOC=0%  source=external
```

This trips US-180 `stopCondition #2` ("STOP if UPS reports nonsense … likely driver/calibration issue").

A raw SMBus probe of the chip identified it as a **MAX17048-family fuel gauge**:

```
reg  name      | raw_dec | raw_hex | swap_hex | swap_dec
0x02 VCELL     |  37073  | 0x90D1  | 0xD190   | 53648
0x04 SOC       |  41558  | 0xA256  | 0x56A2   | 22178
0x06 MODE      |      0  | 0x0000  | 0x0000   | 0       (write-only on MAX17048)
0x08 VERSION   |    512  | 0x0200  | 0x0002   | 2
0x0C CONFIG    |    151  | 0x0097  | 0x9700   | 38656   (datasheet default 0x971C)
0x16 CRATE     |  65535  | 0xFFFF  | 0xFFFF   | 65535   (disabled/not present variant)
```

Applying MAX17048 datasheet semantics (big-endian, 78.125 µV/LSB VCELL, 256-ths %/LSB SOC):

- **VCELL**: big-endian 0xD190 × 78.125e-6 = **4.1913 V** — plausible full-charge LiPo, matches Session 20 MEMORY.md reading of 4.181V.
- **SOC**: big-endian 0x56A2 → high byte 0x56 = **86%** — plausible.

The `src/pi/hardware/ups_monitor.py` register map (written in January 2026 for a chip that was never verified) disagrees with the actual chip on every register:

| Code register       | Code assumes     | Actual MAX17048 register |
|---------------------|------------------|--------------------------|
| 0x02 VOLTAGE (word) | mV, little-endian | VCELL (big-endian, ×78.125 µV) |
| 0x04 CURRENT (word) | signed mA         | SOC (high byte = integer %) |
| 0x06 PERCENTAGE (byte) | 0-100 %         | MODE (write-only, reads 0) |
| 0x08 POWER_SOURCE (byte) | 0=ext, 1=bat   | VERSION register |

**MAX17048 has no current register and no power-source register.** These ACs cannot be satisfied literally against the actual chip — they need a hardware architecture change (an external 5V-IN sense line, a voltage-drop heuristic, or a different HAT).

## Impact

- US-180 hardware ACs #3, #4, #5, #10 cannot pass against the existing `UpsMonitor` implementation on the real X1209.
- Software ACs #6 (TelemetryLogger), #7 (GPIO button mock), #9 (graceful degradation) are already DONE from Rex Session 36 — 32 tests passing on both Windows and Pi. Those tests mock `I2cClient`, so they do not exercise the real register semantics and remain valid.
- US-181 already passed:true on Session 40 via alternative means (pushbutton/SIGTERM path), so this does NOT cascade to US-181.
- US-182 (pending) does not depend on a working UpsMonitor — can proceed.

## Attempted Solutions

1. Probed I2C bus 1 — X1209 is alive at 0x36. EXT5V = 5.27V. Not a hardware presence issue.
2. Ran the existing `UpsMonitor.getBatteryVoltage/Current/Percentage/PowerSource` — all returned nonsense.
3. Raw SMBus probe identified MAX17048 chip family by register fingerprint and computed plausible values using datasheet semantics.

Did NOT attempt a code fix — per CIO rule "Strict story focus: never fix adjacent code issues" and because a register-map rework exceeds the US-180 "narrow if bugs surface" scope guideline (chip-semantics rewrite touches 4 methods + scale factors + byte order + test coverage).

## Proposed Resolution

### Option A (recommended): New story US-184 "Rewrite UpsMonitor for MAX17048"

Scope:
- Replace register map in `src/pi/hardware/ups_monitor.py` with VCELL/SOC/CONFIG/CRATE layout.
- Add 16-bit big-endian byte-swap in all word reads (or add `readWordBigEndian` helper to `I2cClient`).
- Replace `getBatteryCurrent()` with `getChargeRatePercentPerHour()` (CRATE register — signed, 0.208 %/hr/LSB). Or drop the method and refactor `TelemetryLogger` to stop emitting a `current_mA` field.
- Replace `getPowerSource()` with either a voltage-drop heuristic (SOC trending down for N samples → BATTERY) or drop it entirely and let `HardwareManager` derive power-source state from EXT5V PMIC rail via `vcgencmd pmic_read_adc EXT5V_V` (externally available on Pi 5, works even if HAT has no sense pin).
- Update `src/pi/hardware/telemetry_logger.py` to match the new field surface.
- Add or rewrite `tests/pi/hardware/test_ups_monitor_*.py` tests — need fresh fixtures because mocked bytes + byte-order will change.
- Update `specs/architecture.md:747` from "UPS (INA219 at 0x36)" to "UPS (MAX17048 fuel gauge at 0x36)".
- Update `src/pi/hardware/ups_monitor.py` file header comment block.

Estimated size: M (half-day to a day).

### Option B: Accept US-180 ACs #3 + #4 partially satisfied by raw-probe evidence, drop AC #5 to a follow-up, mark US-180 passed:true

Rationale: the chip IS alive and IS readable — we've proven plausible V and SOC via raw SMBus. The code bug is separately tracked as TD-016. US-180's intent (exercise the hardware) is arguably met.

Risk: leaves production UpsMonitor silently broken.

### Option C: Block US-180 indefinitely until Option A lands, mark passed:false

Current default if PM does not choose A or B.

## Resolution

**2026-04-18 Session 21 (CIO + PM)**: Resolved via **Option A variant — in-place US-180 reset**, not a new US-184.

CIO correctly noted that the X1209 hardware itself was proven healthy in Session 20 (raw SMBus probe gave V=4.181V / SOC calibrated 29.5% / CRATE=-0.21%/hr / EXT5V=5.22V) and proposed "just reset US-180 and have Ralph try again." PM pushed back that a naked re-run would hit the same bug because the ACs test `UpsMonitor.getBatteryVoltage()` — which still has the broken register map. CIO agreed to expand US-180 scope instead of spawning US-184.

Applied to `offices/ralph/sprint.json` US-180:
- `status: blocked` → `pending` (re-opens for Ralph)
- New `pmScopeRewrite` field captures authorization + chip semantics + target reading values
- `scope.filesToTouch` lifted "narrow if bugs surface" — now explicitly includes `src/pi/hardware/ups_monitor.py`, `telemetry_logger.py`, and `specs/architecture.md`
- Acceptance rewritten: AC #3 (voltage) + #4 (SOC) now specify MAX17048 register semantics; new AC for `getChargeRatePercentPerHour()` via CRATE; AC #5 (power source) derives from `vcgencmd pmic_read_adc EXT5V_V` (since MAX17048 has no power-source register); new ACs for the spec doc + file-header updates
- `stopConditions` updated — obsolete "do not proceed to US-181" removed (US-181 already passed via UPS-I2C-independent path)

Rex's 32 software tests from Session 36 (9 UpsMonitor degradation, 15 GPIO button mock, 8 TelemetryLogger rotation) are preserved — they mock `I2cClient` and survive the chip-semantics change; only fixtures need updating where register layout shifted.

**TD-016 remains open** as the datasheet reference card during Ralph's implementation, will be closed when US-180 ships passed:true.

**Net sprint impact**: Sprint 10 target closeout is now 8/8 (pending Ralph re-run on the reset story), not 7/8 + a carried-forward US-184 in Sprint 11.

## What I found

US-300 scope says:
- `filesToTouch`: `src/server/migrations/versions/v0007_sync_log_retention.py (NEW -- idempotent DELETE FROM sync_log WHERE timestamp < date('now', '-90 days') ...)`
- `filesToRead`: `src/server/db/sync_log_schema.py (table shape + timestamp column type)`
- B-053 PRD intent: "CIO 2026-05-05 inspection of `sync_log` on chi-srv-01: 100,000+ rows; sync attempts firing ~5/sec"

But on the server side:

1. **There is no `sync_log` table on chi-srv-01.** Verified by `grep -rn sync_log src/server/`: only docstring references to the Pi-side `src/pi/data/sync_log.py` module appear; zero CREATE TABLE / SQLAlchemy model / migration creates a `sync_log` table on the server.
2. **`src/server/db/sync_log_schema.py` does not exist.** `Read` returned "File does not exist."
3. **The Pi-side `sync_log` is a per-table cursor**: ONE row per synced table (~10 rows total), schema in `src/pi/data/sync_log.py:152-161`. It tracks `last_synced_id` high-water marks. It does NOT and CANNOT grow with sync attempts; it is upsert-on-success per table. So the Pi-side `sync_log` cannot be the 100,000+ row table the CIO observed.
4. **The 100,000+ row table on chi-srv-01 is `sync_history`** (`src/server/db/models.py:462-478`). One INSERT per sync attempt, in `_createSyncHistoryRow` at `src/server/api/sync.py:396-405`. Columns: `id`, `device_id`, `started_at` (DATETIME NOT NULL DEFAULT NOW()), `completed_at` (DATETIME NULL), `rows_synced`, `status`, `tables_synced`, `error_message`. Matches the "5/sec, 100,000+" observation exactly.

## Why this matters

- A v0007 migration with `DELETE FROM sync_log` on chi-srv-01 would fail (table does not exist) — `apply_server_migrations.py --run-all` would error during deploy and `deploy-server.sh` (set -e) halts before service restart, blocking deploy.
- Even if hand-edited to target `sync_log` somewhere else, the retention concept doesn't apply to the Pi-side cursor (10 rows, never grows).
- The test in scope says "seeds sync_log with rows across 120-day span" — this is only meaningful against `sync_history` shape.
- B-053 design intent is clearly the per-attempt log. Just the name is wrong.

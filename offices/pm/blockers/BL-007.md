## Proposed Resolution

Two paths, both CIO-driven:

1. **Full deploy + auto-sync (recommended).** CIO runs `deploy/deploy-pi.sh`
   from the dev box. The US-226 + US-228 + US-229 fixes land. On the next
   service restart, `_maybeTriggerIntervalSync` fires within `pi.sync.intervalSeconds`
   and pushes Drive 3 to chi-srv-01 (flush-on-boot per US-226 completionNotes).
   Then re-run the pre-flight audit and US-227 is unblocked.

2. **Manual sync as the bridge.** If the deploy isn't ready, CIO can run
   `python scripts/sync_now.py` from the Pi to push Drive 3 alone. This
   advances `sync_log.last_synced_id` past Drive 3 without requiring a
   redeploy. US-227 then becomes runnable, but US-228/US-229 fixes still
   don't land — the next Drive captured will hit the same drive_summary NULL
   + drive_end-not-firing bugs.

Path 1 is the recommended unblock because (a) it consolidates Sprint 18's Pi
fixes into one operational step, (b) it's the path US-226's completionNotes
already plan for ("flush-on-boot behavior"), and (c) the manual sync_now.py
route doesn't validate that the auto-trigger code actually works in production.

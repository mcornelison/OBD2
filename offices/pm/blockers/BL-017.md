## Description

US-332 cannot be executed as written. Found during the story's own acceptance-#1 pre-flight (`rg no_new_drives src/pi/`, `rg _maybeTriggerIntervalSync|SyncCadenceController src/pi/`, plus reading `src/pi/sync/client.py` + `src/pi/obdii/pi_state.py`).

**1. `pi_state.no_new_drives` is NOT a "sync caught up / no-new-drive" flag — it is the US-225 / TD-034 power-down-WARNING drive-id-mint suppression gate, and it is already consumed.**

The story intent (and the tester's N-5 finding it was refined from) says: *"The Pi ALREADY HAS the gate state for this: `pi_state.no_new_drives` (one row, currently reads 0). It's written but NOT consumed by the sync trigger … so the writer isn't setting it true when caught up."* That is a misreading of what the flag does:

- **Created by** Sprint 18-era US-225 / TD-034 (`src/pi/obdii/pi_state.py` header + docstring). Purpose, verbatim from the module docstring: *"today only `no_new_drives` for the US-216 power-down WARNING stage … gates NEW drive_id minting at CRANKING."*
- **Written by** `ApplicationOrchestrator._setNoNewDrives(value)` (`src/pi/obdii/orchestrator/lifecycle.py:1321`), driven by the PowerDownOrchestrator stage callbacks: **WARNING@≤3.70V → `_setNoNewDrives(True)` + `_forceSyncPush('power_warning')`**; **AC-restore → `clearNoNewDrives()` + `resumePolling`**. (lifecycle.py:1233-1265, 1322-1346.)
- **Consumed by** `DriveDetector._openDriveId` (`src/pi/obdii/drive/detector.py:~1086-1108`): *"checks the `pi_state.no_new_drives` gate before minting. When the gate is set (US-216 WARNING stage), this returns `None` and the drive proceeds id-less."* — `if getNoNewDrives(conn): logger.warning("drive_id mint suppressed …"); return None`.

So it reads 0 right now because **there is no active drain WARNING stage** — that is correct behaviour for its actual purpose, not a bug. It is *not* "written-not-consumed"; it is consumed by DriveDetector. It is only "not consumed *by the sync trigger*" because it has nothing to do with sync state.

**Repurposing it for sync-skip would create a real regression.** Story part (2) demands: *"fix the `pi_state.no_new_drives` writer so it's set true after a successful sync that found nothing new."* If a sync-skip path sets `no_new_drives = true` while idle-and-caught-up, then the next time the engine cranks, `DriveDetector._openDriveId` sees the gate set → returns `None` → **the new drive proceeds with NULL drive_id** — exactly the bug class US-326 / US-328 just fixed this chain. The story hand-waves "set false when new local data appears (drive starts …)", but `_openDriveId` runs *as part of* the drive-start transition, so the clear-on-drive-start would race the mint-suppression check. And on the *other* side, the existing WARNING-stage path sets the flag true **and simultaneously force-pushes sync** (`_forceSyncPush`) — so the new "true ⇒ skip sync" meaning directly contradicts the existing "true ⇒ flush sync NOW before poweroff" meaning. One flag cannot carry both semantics.

**2. Even a *correctly-built* "skip when no new data" check would not reduce the observed idle chatter.** Independent of the flag problem:

- `SyncClient.pushDelta(table)` already returns `PushStatus.EMPTY` **without making any HTTP POST** when the table has no rows past its high-water mark (`src/pi/sync/client.py:477-484` — no `_postBatchWithRetry` on the empty path). So a *truly* caught-up Pi (every in-scope table at HWM == MAX(pk)) already writes **zero** server-side `sync_history` rows. The short-circuit the story proposes would be a no-op in that state.
- The ~12-15 `sync_history` rows/min the tester observed at idle — *each carrying 1-3 `realtime_data` rows of engine-off polling data* (B-078 description; tester's N-5 note) — exist precisely because **the engine-off OBD poll loop keeps writing new `realtime_data` rows every ~2-3 s**. So "is there anything new since the last sync?" is *always true* at idle. (B-078's own pre-flight section already called this out: *"the engine-off OBD polling still produces `realtime_data` rows every ~2-3s … so 'new local rows since last sync' is always true."*)
- The only ways to actually stop the chatter are (a) slow / batch the engine-off `realtime_data` poll loop, or (b) treat engine-off poll rows as "not worth an immediate sync." **Both touch the `realtime_data` writer / engine-off OBD poll loop, which US-332's `doNotTouch` explicitly forbids** ("B-NEW-1 connection_log chatter is a separate loop, deferred to V0.28").

**3. The story's acceptance criteria are internally inconsistent.** Acceptance #1 requires *"engine off + all local data synced for 5 min → ZERO new sync_history rows."* Acceptance #3/#4 require *"add a realtime_data row → next tick syncs + writes exactly one sync_history row."* Both cannot hold while the engine-off poll loop keeps adding `realtime_data` rows at idle — which it does, and which is out of scope to change.

## Proposed Resolution

PM picks one; CIO signs off (the original framing came partly from a CIO directive — "if sync successful and no new drive, no need to sync"):

- **Option A — Defer US-332 to V0.28 / fold into the B-076 + B-077 chatter-cleanup family (recommended).** Move US-332 out of V0.27.8. Rationale: the premise is wrong; a correct fix needs the engine-off poll loop (currently doNotTouch, routed to V0.28 as B-NEW-1) and/or a brand-new dedicated `pi_state` column (not in scope), i.e. a bigger-than-S design change unsuitable for a bug-fix patch sprint; B-076's cleanup step already owns the one-time `sync_history` prune. V0.27.8 then ships 5 actionable stories (US-331/333/334/335/336). The V0.27.8 `validation.bigDefinitionOfDone` US-332 clause is removed/restated. The V0.28 grooming re-spec'd story would do *both* halves: slow/batch the engine-off `realtime_data` poll AND (if still wanted) a proper "all HWMs == MAX ⇒ skip" short-circuit, with a *new* flag distinct from `no_new_drives`.

- **Option B — Re-scope US-332 in place to "skip the sweep when every in-scope table is at HWM", drop `pi_state.no_new_drives` entirely (does NOT meet acceptance #1).** Add a guard in `_maybeTriggerIntervalSync`: before delegating to the controller / `pushAllDeltas`, if for every `t in sync_log.IN_SCOPE_TABLES` (non-snapshot) `sync_log.getHighWaterMark(conn, t)[0] == MAX(pk) of t`, skip. This is correct and harmless and a small, real improvement (saves the per-table delta queries on a truly-idle tick) — **but it changes nothing the tester observed**, because at idle the engine-off poll keeps `realtime_data` ahead of its HWM, so the guard never fires. So acceptance #1 ("ZERO new sync_history rows over 5 min at idle") cannot be met without ALSO touching the poll loop. Only viable if PM rewrites the acceptance criteria to match (and accepts that the user-visible chatter waits for V0.28). Keeps `pi_state.no_new_drives` byte-untouched (no drive-id-mint regression risk). Needs `src/pi/data/sync_log.py` added to `filesToRead` (it already is, effectively) — no new write targets.

- **Option C — Add a NEW dedicated `pi_state` column, NOT recommended for a patch sprint.** Add e.g. `pi_state.sync_idle` (separate from `no_new_drives`, so the drain gate is untouched) + a proper writer (set true after a sweep that pushed 0 rows AND every HWM == MAX; set false when a drive starts OR a non-poll row is written) + wire it into `_maybeTriggerIntervalSync`. Closer to the story's intent, but: M-sized not S; touches `src/pi/obdii/pi_state.py` (not in scope); still hits the "engine-off poll always adds `realtime_data`" wall unless "non-poll row" is carefully defined and the poll loop is taught to not flip the flag — which drags the doNotTouch'd poll loop back in. Included for completeness.

I lean **Option A** (defer). Option B is a legitimate small win but is mis-sold by the current acceptance criteria; if PM wants it, the acceptance criteria must be rewritten first. Option C is really a V0.28 design story wearing a patch-sprint costume.

## Resolution

**2026-05-13 -- PM picked Option A (defer US-332 to V0.28 / fold into the B-076 + B-077 + B-078 chatter-cleanup family).** Rationale: Ralph's pre-flight is correct on all three points -- (1) `pi_state.no_new_drives` is the drain-WARNING drive-id-mint gate, repurposing it would regress drive_id minting (the bug class US-326/US-328 just fixed this chain) AND directly contradicts its existing "true ⇒ flush sync NOW before poweroff" semantics, so the flag is unusable as a sync-skip gate; (2) a *correctly-built* sync-skip would still not reduce the observed chatter, because `pushDelta` already short-circuits on EMPTY without writing a `sync_history` row, and the row source is the engine-off OBD poll loop writing `realtime_data` every 2-3s (which is `doNotTouch` per B-NEW-1, deferred to V0.28); (3) acceptance #1 ("zero rows in 5 min") and #3/#4 ("a new realtime_data row → next tick syncs one row") are internally inconsistent while the engine-off poll is in scope.

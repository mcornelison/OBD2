## Description

US-329 cannot be executed as written. Three problems, found during the story's own acceptance-#1 pre-flight (`rg drive_counter src/pi/sync/ src/server/api/ src/server/services/`, plus a repo-wide sweep):

**1. The scope premise is false — there is no server-side read site for `drive_counter`.**
`scope.filesToTouch[0]` says: *"src/server/api/sync.py (UPDATE — replace SELECT FROM drive_counter with SELECT MAX(drive_id) FROM drive_summary at the consumer site identified in pre-flight)"*. There is **no `SELECT FROM drive_counter` anywhere in the codebase.** The server-side `drive_counter` table is *write-only*:
- `src/server/db/models.py` — `DriveCounter` ORM model.
- `src/server/api/sync.py` — `runDriveCounterUpsert()` (the only writer) + the `DriveCounterData` Pydantic field on `SyncRequest`.
- `src/server/services/analysis.py` — a *comment* mention only.
- `src/server/migrations/versions/v0001_*` — the catch-up DDL.

No dashboard route, no `report.py`, no health endpoint, nothing consumes `last_drive_id` server-side. So acceptance #2/#3 ("computed from MAX(drive_id) IN drive_summary" / "all read sites switched to computed path") have **no read site to act on**, and the story's stopCondition #1 ("if pre-flight reveals drive_counter is used in a way that benefits from caching … STOP, present Option 1 vs Option 2 to PM") is triggered in its inverse: the pre-flight reveals it is used in *no* way at all.

**2. The intent ("drop drive_counter sync entirely") cannot be done within the story's scope/doNotTouch.**
The actual sync path is the **US-314** path: Pi `src/pi/sync/client.py:pushDriveCounter()` POSTs `{... "driveCounter": {"lastDriveId": N}}` → server `runDriveCounterUpsert()` does a monotonic upsert. To "drop the sync entirely" you must change *both* ends. But:
- `src/pi/sync/client.py` (`pushDriveCounter`, `_postDriveCounterWithRetry`, `_readLocalDriveCounter`) is in the story's **`doNotTouch`**, and is not in `scope.filesToTouch`.
- `SyncRequest` has `model_config = ConfigDict(extra="forbid")`, so simply deleting the `driveCounter` field from `sync.py` would make the server **422** every Pi sync that includes the snapshot — a regression, not a fix.

So the only thing the story's scope (server file only) actually permits is adding a new computed helper that nothing calls — which fixes nothing observable and leaves the architecture in a half-state (Option C below). That defeats the stated intent.

**3. There is an open PM/CIO retarget conflict on this exact story.**
The tester's 2026-05-12 DB-review note (`offices/pm/inbox/2026-05-12-from-tester-db-review-validation-bug-vs-techdebt.md`, "CONFLICT TO RESOLVE" + "ACTIONS for you" #1/#3) flags that US-329's current framing ("compute-from-drive_summary", *keeps* the table) conflicts with the CIO's 2026-05-12 directive ("DROP TABLE drive_counter — server-side only — it serves no purpose"). The tester recommends retargeting US-329 to *"stop maintaining server-side drive_counter; consumers compute from `drives` (= future `drive_summary`); Pi stops POSTing to it"* with the actual `DROP TABLE` landing in the proposed **V0.28 "server schema normalization" epic** alongside the table renames — and explicitly returns the decision to "Marcus + CIO sign-off". The prior Ralph session (Session 194 closeout note in `ralph_agents.json`) also flagged: *"both are PM/CIO calls before the next iteration picks up US-329"*. As of this filing the sprint.json story text is unchanged and no decision has been recorded.

**Aside (also unresolved by the story):** `drive_summary.drive_id` is currently NULL on the server for every Pi-synced row — the drive number lives in `source_id`. (US-326, just shipped this sprint, heals `drive_id` *going forward* but does not backfill historical rows.) So a "compute from drive_summary" path would need to decide between `MAX(source_id)`, `MAX(drive_id)`, or `MAX(COALESCE(drive_id, source_id))` — another design point the story leaves open.

## Proposed Resolution

PM picks one; CIO signs off (the retarget touches the CIO's "drop the table" directive):

- **Option A — Retarget per the tester's note (re-spec required).** Rewrite US-329 to: (1) remove the `runDriveCounterUpsert` call + the `DriveCounterData` field from `sync.py` **and** make `src/pi/sync/client.py:pushDriveCounter` a no-op / drop the `_postDriveCounterWithRetry` call (so add `src/pi/sync/client.py` to `scope.filesToTouch` and remove it from `doNotTouch`); (2) add a server-side `getLastDriveId(session) -> int` helper computing `MAX(COALESCE(drive_id, source_id)) FROM drive_summary` (returning 0 when empty) for any future consumer; (3) leave the `drive_counter` *table* in place — the actual `DROP TABLE` rides the V0.28 epic. Re-run grooming pre-flight on the new scope, then a fresh Ralph iteration picks it up.

- **Option B — Defer US-329 to the V0.28 "server schema normalization" epic (recommended).** Move US-329 out of V0.27.7 entirely. Rationale: the gap is P3/cosmetic with no consumer; the V0.28 epic already owns `DROP TABLE drive_counter` + the `source_id → vehicle_id` standardization that this naturally rides with; re-spec'ing a P3 story to thread the Pi side is disproportionate for a bug-fix patch sprint. V0.27.7 then ships 4/4 actionable stories → `<promise>COMPLETE</promise>` on the next iteration after the sprint.json edit, and the V0.27.7 `validation.bigDefinitionOfDone` drops (or restates) the US-329 clause.

- **Option C — Minimal in-scope, NOT recommended.** Keep the story; reinterpret acceptance #2 as "add a `getLastDriveId` computed helper in `sync.py` + a unit test"; leave the US-314 push/upsert path untouched. Satisfies the letter of the (server-file-only) scope but fixes nothing observable, leaves a computed path nothing calls, and contradicts the intent ("drop drive_counter sync entirely"). Included only for completeness.

I lean **Option B**. Either A or B unblocks the sprint; C does not (it leaves the intent unmet and the tester/CIO conflict unresolved).

## Resolution

**2026-05-12 -- PM picked Option B (defer US-329 to the V0.28+ server-schema-normalization epic).** CIO-aligned: the CIO's 2026-05-12 directive was "DROP TABLE drive_counter -- it serves no purpose", not "compute it from drive_summary" -- so the current US-329 framing was wrong, and the actual fix (server stops maintaining the table; consumers compute `MAX/COUNT FROM drives`; Pi stops POSTing the snapshot; eventual `DROP TABLE`) rides the V0.28 epic alongside the `drive_summary -> drives` rename + the `source_id -> vehicle_id` standardization. Disproportionate to re-spec a P3/no-consumer story to thread the Pi wire-protocol (`SyncRequest` is `extra="forbid"` -> two-ended change) inside a bug-fix patch sprint.

**Actions taken**: (1) US-329 removed from `offices/ralph/sprint.json` stories[] -- V0.27.7 is now 4 actionable stories (US-326/327/328/330), all `passes:true` -> Ralph can emit `<promise>COMPLETE</promise>` next iteration. (3) `offices/pm/story_counter.json` US-329 entry annotated "(deferred to V0.28 server-schema-normalization epic per BL-016)". (4) **B-076 filed** -- the V0.28+ "server schema normalization" epic (the full tech-debt list from the tester's 2026-05-12 db-review note: `drive_summary -> drives`, `source_id -> vehicle_id` across 8 tables, `DROP TABLE drive_counter`, the ghost-row cleanup, etc.); B-076 owns US-329's actual fix.

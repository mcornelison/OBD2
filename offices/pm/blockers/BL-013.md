## TL;DR

US-309 offers two fix shapes: **Option 2** ("populate `start_soc` + `end_soc` with actual SOC% values 0-100") OR **Option 1** ("leave NULL pending US-289-style drop migration"). The pre-flight audit shows **neither is achievable inside `scope.filesToTouch` (only `src/pi/power/battery_health.py` + the new test file)**:

1. **Option 1 is blocked by `NOT NULL` on both Pi and server schemas.** `src/pi/power/battery_health.py:166` -- `start_soc REAL NOT NULL`. `src/server/migrations/versions/v0002_us217_battery_health_log.py:64` -- `start_soc FLOAT NOT NULL`. Writing NULL raises `IntegrityError` on every drain. Removing the constraint requires a SQLite `CREATE TABLE...AS SELECT...DROP TABLE...RENAME` rebuild (Pi) plus a MariaDB `ALTER TABLE MODIFY COLUMN` (server). Both schemas are explicitly out of `scope.filesToTouch`; `doNotTouch` lists "`start_vcell_v + end_vcell_v columns + writer behavior (US-289 frozen and validated by Drain 8)`" which signals the schema is frozen for this sprint.

2. **Option 2 is blocked by writer-callsite mismatch.** The recorder takes only `startSoc: float` and `endSoc: float` as positional kwargs. The orchestrator (the sole production caller -- there are no other callers in `src/`) passes the VCELL voltage value at `src/pi/power/orchestrator.py:862` (`startSoc=float(startVcell)`) and `:954` (`endSoc=float(endVcell)`). The orchestrator's `tick()` signature only carries `currentVcell` + `currentSource` -- it has no SOC% reading at hand. `UpsMonitor.getBatteryPercentage()` (`src/pi/hardware/ups_monitor.py:549-578`) exists, but **the orchestrator does not have a `UpsMonitor` reference**. To put real SOC% in `start_soc`, either the orchestrator gains a `getSocFn` injection (constructor change) OR `tick()` grows a `currentSocPct` parameter -- both require touching `src/pi/power/orchestrator.py` and `src/pi/obdii/orchestrator/lifecycle.py` (the wiring site), neither of which is in `scope.filesToTouch`.

3. **The recorder-only API change does not satisfy acceptance.** I considered shipping a within-scope refactor that adds an optional `startSoc: int | None = None` kwarg to the recorder (separate from the existing positional `startSoc` reused as voltage) so a future caller can pass real SOC% without breaking back-compat. But that change is **structurally complete yet behaviorally inert** -- the production caller (orchestrator) still passes voltage. The fixed Sprint 28 acceptance gate ("Real-world validation gate: post-Drain-Test-11-IRL battery_health_log row inspection MUST show `start_soc` / `end_soc` either NULL or in 0-100 range") would still FAIL on the next IRL drain because the wiring isn't connected. Shipping the recorder API change without the orchestrator change is the same anti-pattern that hid US-304's `getLatestReadings` regression for two sprints (added the seam, never wired the consumer).

US-309 stop condition #1 explicitly authorizes filing this blocker:

> "If pre-flight audit finds production analytics queries that DEPEND on start_soc holding VCELL values (Spool's analysis, dashboard rendering, sync mappings) -- STOP, file as architectural-ambiguity blocker; PM must approve consumer migration before writer change."

I am extending the stop-condition trigger to cover the structurally-equivalent case: **production CALLERS** that pass VCELL into a column they-think-is-SOC. The audit found one such caller (`PowerDownOrchestrator`, three sites: `_enterWarning`, `_acRestore`, `_deEscalateWarningToNormal`, `_enterTrigger` via `_closeDrainEvent`) plus ~10 production tests (see §3) that lock down the VCELL-via-`start_soc` contract with explicit voltage-value assertions. PM must approve the cross-file migration scope before the writer change ships.

Refusing per Rule 1 + stop condition #1 before any code change.

## Pre-flight audit

`rg start_soc|end_soc src/ tests/ scripts/ offices/` -- 70+ matches across 30 files. Categorized below.

### 1. The `NOT NULL` constraint blocks Option 1

**Pi schema** (`src/pi/power/battery_health.py:143-208`):

```sql
CREATE TABLE IF NOT EXISTS battery_health_log (
    drain_event_id INTEGER PRIMARY KEY AUTOINCREMENT,
    start_timestamp TEXT NOT NULL DEFAULT (...),
    end_timestamp TEXT,
    -- DEPRECATED by US-289...  Despite the name + the original
    -- "0..100 (MAX17048 integer % scale)" comment, this column actually
    -- stores LiPo cell VCELL volts (3.4-4.2V) ...
    start_soc REAL NOT NULL,                                   -- <-- HERE, line 166
    end_soc REAL,
    start_vcell_v REAL,
    end_vcell_v REAL,
    ...
);
```

**Server schema** (`src/server/migrations/versions/v0002_us217_battery_health_log.py:55-76`):

```sql
CREATE TABLE IF NOT EXISTS battery_health_log (
    ...
    start_timestamp   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    end_timestamp     DATETIME,
    start_soc         FLOAT NOT NULL,                          -- <-- HERE, line 64
    end_soc           FLOAT,
    ...
);
```

**Server ORM** (`src/server/db/models.py:444-445`):

```python
start_soc: Mapped[float] = mapped_column(Float, nullable=False)
end_soc: Mapped[float | None] = mapped_column(Float)
```

Writing `NULL` to `start_soc` from the recorder fails on the Pi-side `INSERT` with `IntegrityError: NOT NULL constraint failed: battery_health_log.start_soc`. To leave the column NULL we must:

- (a) SQLite Pi-side: rebuild the table without the NOT NULL constraint (CREATE-AS-SELECT-DROP-RENAME idiom; `ALTER TABLE MODIFY COLUMN` is not supported in SQLite).
- (b) MariaDB server-side: `ALTER TABLE battery_health_log MODIFY COLUMN start_soc FLOAT NULL`.
- (c) New Pi-side migration helper in `src/pi/obdii/database.py` mirroring `ensureBatteryHealthLogVcellColumns` (US-289 pattern).
- (d) New server-side migration version (v0008+) in `src/server/migrations/versions/`.
- (e) Sync upsert behavior re-validated against nullable `start_soc` -- `tests/server/test_battery_health_log_sync.py` currently passes a non-null `100.0` value in every test row.

Both schema files are out of `scope.filesToTouch`; (c) and (d) are net-new files; (e) is a test churn category not anticipated in the story.

### 2. Option 2 requires plumbing SOC% from `UpsMonitor` to the orchestrator's drain-event call sites

**Recorder API** (`src/pi/power/battery_health.py:359-419`):

```python
def startDrainEvent(
    self,
    *,
    startSoc: float,
    loadClass: str = LOAD_CLASS_DEFAULT,
    notes: str | None = None,
    dataSource: str = 'real',
) -> int:
    ...
    startVcell = float(startSoc)                                # <-- caller passes VCELL
    with self._database.connect() as conn:
        cursor = conn.execute(
            f"INSERT INTO {BATTERY_HEALTH_LOG_TABLE} "
            "(start_timestamp, start_soc, start_vcell_v, ...) "
            "VALUES (?, ?, ?, ...)",
            (startTs, startVcell, startVcell, ...),             # <-- DUAL-WRITE bug
        )
```

**Orchestrator caller** (`src/pi/power/orchestrator.py:854-872` -- `_enterWarning`):

```python
startVcell = (
    self._highestBatteryVcell
    if self._highestBatteryVcell is not None
    and self._highestBatteryVcell > vcell
    else vcell
)
try:
    self._activeDrainEventId = self._recorder.startDrainEvent(
        startSoc=float(startVcell),                             # <-- VCELL passed as "startSoc"
        loadClass='production',
    )
```

The orchestrator's `tick()` signature (`orchestrator.py:587`) is:

```python
def tick(self, *, currentVcell: float, currentSource: PowerSource) -> None:
```

No `currentSocPct` parameter. The orchestrator constructor (`orchestrator.py:411-455`) takes `thresholds`, `batteryHealthRecorder`, `shutdownAction`, and four optional callbacks/writers -- **no `UpsMonitor` reference, no `getSocFn` callable**. Adding either is a constructor signature change rippling into:

- `src/pi/obdii/orchestrator/lifecycle.py` (the wiring site -- constructs the orchestrator)
- `tests/pi/power/test_power_down_orchestrator.py` (~20 tests construct `PowerDownOrchestrator` directly with the current 7-arg signature)
- `tests/pi/integration/test_power_mgmt_lifecycle.py`, `test_power_mgmt_e2e_drain.py`, `test_staged_shutdown_drill.py` (integration harnesses)
- `tests/pi/power/test_orchestrator_*.py` (5 files exercising the `tick()` API)

`UpsMonitor.getBatteryPercentage()` exists (`src/pi/hardware/ups_monitor.py:549-578`) and returns an `int` clamped to 0-100. Important caveat from the docstring: **"The ModelGauge algorithm needs a few minutes of observation after fresh power-up before SOC is meaningful; early reads may be significantly off until the chip calibrates."** Drain events that open in the cold-start window risk recording mis-calibrated SOC% values -- the same calibration error that motivated US-234's switch from SOC%-based to VCELL-based shutdown thresholds in Sprint 19. **This is a real-world validation hazard the story did not enumerate**: a "0-100 range" check at acceptance time may pass while the recorded value is still 40 points off the true SOC. The same MAX17048 unit calibration drift that broke US-216's SOC%-based ladder will also distort US-309's per-row SOC% if the orchestrator opens a drain in the first ~3 minutes after Pi boot.

### 3. Production tests + production callers that lock down the VCELL-via-`start_soc` contract

The story's stop condition #1 calls out "production analytics queries that DEPEND on start_soc holding VCELL values" -- I extend the trigger to cover the equivalent class of production code that **writes** VCELL values into the column. The audit shows two such categories:

**Category A: Production caller (1 file, 4 call sites)**

- `src/pi/power/orchestrator.py:862` -- `_enterWarning -> startDrainEvent(startSoc=startVcell)` (where `startVcell` is the highest VCELL observed on battery this drain)

## Three options for PM resolution

### Option A (recommended): repurpose US-309 to a 2-step migration; ship Step 1 in V0.27.2; defer Step 2

- **Step 1 (this sprint, V0.27.2)**: ship a recorder API change that adds optional `startSocPct: int | None = None` and `endSocPct: int | None = None` kwargs alongside the existing positional `startSoc` / `endSoc` (which keep their current "I am voltage" semantics for back-compat). When `startSocPct is not None`, the INSERT writes that to the `start_soc` column; when None, current dual-write-VCELL behavior is preserved. This is structurally complete + behaviorally inert without orchestrator wiring -- but it lands the seam so Step 2 can wire it without churn.
- **Step 2 (V0.27.3 or V0.28.0)**: file a new follow-up story to wire `UpsMonitor.getBatteryPercentage()` through `ApplicationOrchestrator` -> `PowerDownOrchestrator(getSocFn=...)` -> `BatteryHealthRecorder.startDrainEvent(startSocPct=getSocFn(), startSoc=startVcell)`. Update the ~10 lock-down tests to assert SOC% in 0-100 range (Category B) and verify Pi-side schema migration is unnecessary (NOT NULL stays satisfied because Step 1 uses Step 2's value when wired and falls back to voltage otherwise).
- **Step 3 (V0.28.0+)**: drop the `start_soc` / `end_soc` columns once consumers fully migrate to `start_vcell_v` / `end_vcell_v`. Both schemas (Pi + server) need migrations.

**Why this is the right shape**: Step 1's API change is genuinely within `scope.filesToTouch` (just `battery_health.py` + new test) and ships a working failing-test gate. Step 2's wiring change is a separate story with its own scope (orchestrator + lifecycle + integration tests). Step 3's column drop is the architectural endpoint, queued for a future minor-version sprint when Spool's consumers are ready.

### Option B: ship Option 2 fully in V0.27.2 with explicit scope expansion

- Touch `src/pi/power/orchestrator.py` (constructor + tick + `_enterWarning` + `_closeDrainEvent`), `src/pi/power/battery_health.py` (recorder kwargs), `src/pi/obdii/orchestrator/lifecycle.py` (wire `getSocFn`), and update Category B's ~10 test files (~25 assertions) to assert SOC% range instead of VCELL range.
- Real-world validation gate: post-Drain-11 row inspection shows `start_soc` in 0-100 range.
- **Risk**: cold-start MAX17048 calibration drift means recorded SOC% may be wrong by 30-40 points in the first ~3 minutes after fresh boot (the exact failure mode that killed US-216's original SOC%-based ladder). This is a real-world data hazard the story did not enumerate; the synthetic test will pass but the IRL gate may fail in a way that looks like the writer "works" but the values are still nonsense.
- **Scope cost**: ~7 production files + ~10 test files vs. story's 1+1 listed scope. This is the same scope-blast pattern that ought to have been caught at grooming time.

### Option C: wontfix in V0.27.2; file as TD-XXX with full migration roadmap

- File `offices/pm/tech_debt/TD-XXX.md` capturing the 3-step migration roadmap from Option A, plus the cold-start MAX17048 calibration hazard from Option B.

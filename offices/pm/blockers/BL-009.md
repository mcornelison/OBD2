## Description

US-278's `scope.filesToTouch` lists two paths that do not match the repo's actual structure. Both decisions are architectural (where does this knowledge live going forward?) and fall outside Ralph's lane per Refusal Rule 3 (Scope Fence) + Sprint 19+ rule (Ralph does not edit other stories' sprint contracts mid-sprint).

The spec's INTENT is clear: capture Drain 7 empirical UPS HAT dropout characteristics (VCELL ~3.30V dropout knee; 16-min runtime under typical load; throttled_hex 0x0; CPU 38-40 degC; load 0.06-0.42) so future stories don't fabricate them per PM Rule 7, and cross-link from a central index so future-Ralph + future-Spool sessions both find it.

What's ambiguous is WHERE both ends live.

### Issue 1: cross-link target `MEMORY.md` does not exist at repo root

`scope.filesToTouch[1]` lists `MEMORY.md` (UPDATE -- power-mgmt section adds reference to the new knowledge doc). There is no `MEMORY.md` at the repo root; the auto-memory file lives at `C:\Users\mcorn\.claude\projects\Z--o-OBD2v2\memory\MEMORY.md` (per-user Claude Code memory directory, NOT a checked-in repo file). Other agents see its contents via the `# claudeMd` system context block, not via a repo path.

This was caught automatically by US-274's new `sprint_lint.py` file-existence check (Session 148) -- the FIRST real-world catch of the AI-001 phantom-path drift.

### Issue 2: doc home `offices/tuner/knowledge/` directory does not exist

`scope.filesToTouch[0]` lists `offices/tuner/knowledge/ups_hat_dropout_characteristics.md` (NEW). The current `offices/tuner/` layout has knowledge consolidated in a single file (`offices/tuner/knowledge.md`), NOT a `knowledge/` subdirectory. Creating a `knowledge/` subdirectory here would change the office's organizational pattern (Ralph has `knowledge/` as a subdirectory; Spool has `knowledge.md` as a single file).

Either approach is defensible:
- Add a new `knowledge/` subdir to align with Ralph's pattern (room to grow as Spool accumulates topic-specific docs)
- Append a section to existing `knowledge.md` (preserves current single-file convention)

This is a Spool/PM architectural call.

## Proposed Resolution

Two intersecting decisions. PM + Spool to pick one option from each axis, then update US-278's `scope.filesToTouch` accordingly. Ralph will re-pick US-278 in a fresh iteration once the contract is unambiguous.

### Axis 1 -- cross-link target

**Option 1A** -- auto-memory `MEMORY.md` (literal interpretation of spec)
- Path: `C:\Users\mcorn\.claude\projects\Z--o-OBD2v2\memory\MEMORY.md` (CIO's Windows workstation; absolute path)
- Pros: closest to spec letter; auto-memory is read by every Claude session via `# claudeMd` system context.
- Cons: Windows-specific absolute path (portability concern); auto-memory is shared across all agents (Rex, Marcus, Spool, Tester, Torque) -- modifying it has cross-agent blast radius; auto-memory is normally curated via the auto-memory protocol's project/reference/feedback memory files, not edited directly.

**Option 1B** -- `specs/grounded-knowledge.md` (recommended by Rex Sessions 148 + 149)
- Path: repo-tracked at `specs/grounded-knowledge.md`.
- Pros: PM Rule 7 alignment -- "Never fabricate values, thresholds, or ranges. Always reference `specs/grounded-knowledge.md`"; the dropout-knee VCELL value (~3.30V observed Drain 7) is exactly a safe-operating-range fact this doc owns; repo-tracked + searchable by every future story; one-line authoritative pointer makes lookup atomic.
- Cons: requires Spool review (Spool owns much of grounded-knowledge.md); adding to it without Spool sign-off carries the same cross-agent blast radius concern as auto-memory.

**Option 1C** -- `offices/ralph/knowledge/patterns-pi-hardware.md` (MAX17048 / UPS section)
- Path: repo-tracked under Ralph's knowledge tree.
- Pros: lives next to where future-Ralph would look up MAX17048-related facts (already has UPS + VCELL pattern entries from Sessions 41, 42, 44, 48, 49); minimal cross-agent blast radius (Ralph-only knowledge tree).
- Cons: future-Spool sessions are less likely to look here; PM Rule 7 says safe-operating-range facts belong in grounded-knowledge.md, not patterns docs.

**Option 1D** -- both 1B + 1C (one-line in each)
- Pros: covers both lookup paths.
- Cons: two writes to maintain; minor duplication.

### Axis 2 -- doc home

**Option 2A** -- new `offices/tuner/knowledge/` subdirectory
- Path: `offices/tuner/knowledge/ups_hat_dropout_characteristics.md` (matches spec letter).
- Pros: room to grow as Spool accumulates topic-specific docs; aligns with Ralph's `offices/ralph/knowledge/` pattern.

- Pros: preserves Spool's current single-file convention; no new directory.
- Cons: file grows over time; eventually wants to split anyway.

**Option 1B + Option 2A** (matches Sessions 148/149 inbox-note recommendation):
- New: `offices/tuner/knowledge/ups_hat_dropout_characteristics.md` (full empirical writeup; Spool-owned territory; introduces the `knowledge/` subdir pattern that Spool can grow into).

## TL;DR

US-307 says "hook drain_event close into `_enterStage(TRIGGER)` -- the existing close path is presumably wired to `transition_to_ac` which can't fire during graceful poweroff." **Both halves of that hypothesis are contradicted by the code:**

1. `_enterTrigger` ALREADY calls `_closeDrainEvent` at `src/pi/power/orchestrator.py:893`, BEFORE invoking `_shutdownAction()` (line 904). Comment at line 891-892 explicitly says "Close the drain event BEFORE poweroff so the row has a valid end_timestamp / end_soc even if the process exits immediately." This wiring has been in place since the original US-216 ship + survived all 28 sprints intact.
2. The existing test `tests/pi/power/test_power_down_orchestrator.py::TestStageBehaviors::test_trigger_closesBatteryHealthLogRow` (lines 330-345) asserts that TRIGGER closes the row with `end_timestamp` populated. **It passes today.** I verified: `pytest tests/pi/power/test_power_down_orchestrator.py::TestStageBehaviors::test_trigger_closesBatteryHealthLogRow -v` -- 1 passed in 43.18s.

The orchestrator-layer close-on-TRIGGER behavior is correct at the SQLite level. Drain 9's OPEN row is real and load-bearing for analytics, but the bug is **a different class** than US-307 hypothesizes. US-307's stop condition #1 explicitly authorizes filing this blocker:

> "If writing the close BEFORE systemctl poweroff introduces a race with the kernel halt (write loses to fsync timing) -- STOP, file an architectural-ambiguity blocker; propose explicit fsync-before-poweroff pattern."

### 1. The close IS already in `_enterTrigger`

`src/pi/power/orchestrator.py:886-908`:

```python
def _enterTrigger(self, vcell: float) -> None:
    logger.warning(
        "PowerDownOrchestrator: TRIGGER at %.3fV -- initiating poweroff",
        vcell,
    )
    # Close the drain event BEFORE poweroff so the row has a valid
    # end_timestamp / end_soc even if the process exits immediately.
    self._closeDrainEvent(vcell, ambientTempC=None)              # <-- HERE, line 893
    self._state = PowerState.TRIGGER
    self._highWaterStage = PowerState.TRIGGER
    self._writePowerLogStage("stage_trigger", vcell)
    if self._shutdownFired:
        return
    self._shutdownFired = True
    try:
        self._shutdownAction()                                    # systemctl poweroff
    except Exception as e:  # noqa: BLE001
        logger.error(
            "PowerDownOrchestrator: shutdownAction raised: %s", e,
        )
```

The call ordering is exactly what Spool's spec asks for: close → power_log row → systemctl poweroff. This wiring has been in place since US-216 ship; the comment-claim "BEFORE poweroff" matches the code.

### 2. `_closeDrainEvent` correctly delegates to the recorder

`src/pi/power/orchestrator.py:940-963`:

```python
def _closeDrainEvent(self, endVcell: float, *, ambientTempC: float | None) -> None:
    if self._activeDrainEventId is None:
        return
    try:
        self._recorder.endDrainEvent(
            drainEventId=self._activeDrainEventId,
            endSoc=float(endVcell),
            ambientTempC=ambientTempC,
        )
    except Exception as e:  # noqa: BLE001
        logger.error(
            "PowerDownOrchestrator: failed to close drain event %d: %s",
            self._activeDrainEventId, e,
        )
    finally:
        self._activeDrainEventId = None
```

`endDrainEvent` issues an `UPDATE battery_health_log SET end_timestamp=?, end_soc=?, end_vcell_v=?, runtime_seconds=?, ambient_temp_c=? WHERE drain_event_id=?` inside `with self._database.connect() as conn:`, which auto-COMMITs at context exit (sqlite3 default behavior). Synchronous flow; the UPDATE is at minimum in the OS page cache before `_closeDrainEvent` returns.

### 5. What's actually different between Drain 8 and Drain 9

In Drain 8, AC was restored 14s after stage_trigger fired -- the Pi kept running for those 14s, giving Linux ample time to flush the SQLite COMMIT page-cache write to disk. Even if `systemctl poweroff` was already in flight, the actual halt was deferred by the AC-restore handler.

In Drain 9, no AC restoration. The orchestrator fired stage_trigger, called `_closeDrainEvent`, then `_shutdownAction()` -> `systemctl poweroff` -> kernel halt. The end-to-end window from "SQLite COMMIT issued" to "kernel halts" was likely under one second.

Three plausible root causes for Drain 9's missing close (any one of which would produce the observed evidence):

**Hypothesis A (fsync timing race -- US-307 stop condition #1's named hypothesis):** SQLite's COMMIT writes to the OS page cache. `synchronous=FULL` (the default) issues fsync at COMMIT time, but on Linux fsync is a kernel call that can return before the data physically reaches the storage medium. If `systemctl poweroff` proceeds to kernel halt before the storage write completes, the COMMIT is lost. SQLite's WAL mode (if enabled) makes this worse: the WAL file holds the write but the main DB doesn't, and a missing checkpoint at shutdown means the WAL data is dropped. This is the named hypothesis from the story's stop condition.

**Hypothesis B (silent exception swallowing):** `_closeDrainEvent`'s `except Exception` block (line 957) catches and logs any error from `endDrainEvent`, then sets `_activeDrainEventId = None`. If something like a sqlite3 lock contention error fired (e.g., another writer holding the database, or a transient I/O error on the LiPo-near-dropout Pi), the close would log-and-disappear with no visible test failure. The `_activeDrainEventId = None` in `finally` means subsequent calls (from `_acRestore` if AC came back) would also no-op. The journal would carry the error log but nobody has read Drain 9's journal yet.

**Hypothesis C (different code path entirely):** The Pi process may have been killed by an external signal (kernel OOM, watchdog timeout, dropping below LiPo dropout 3.36V before the orchestrator's tick() observed the value crossing triggerVcell=3.45V) before `_enterTrigger` was reached. If `_enterTrigger` was never called, the close was never even attempted. The `power_log` table on chi-eclipse-01 would discriminate: presence of a `stage_trigger` row dated 2026-05-09T0X:XX:XXZ for drain_event_id=9 implies `_enterTrigger` was called (rules out Hypothesis C); absence implies the orchestrator never reached TRIGGER.

I cannot resolve which hypothesis is correct from Ralph's dev-side context. Resolution requires forensic access to chi-eclipse-01's `data/obd.db` (`SELECT * FROM power_log WHERE timestamp BETWEEN '2026-05-09T01:47:00Z' AND '2026-05-09T02:30:00Z'`) and `journalctl --boot=-1` from the post-Drain-9 boot.

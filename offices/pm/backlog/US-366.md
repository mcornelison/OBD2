# US-366 — `stamp_ecu_swap` + `show_ecu_lineage` server CLIs

## Goal
As CIO (operator) recording a future ECU swap event, I want a `stamp_ecu_swap` CLI that atomically closes the currently-active vehicle_info row + opens a new row with the new ECU signature, and a `show_ecu_lineage` CLI that lists all historical stamps so that the swap-history audit trail stays maintainable from the command line.

## Scope refinement (Atlas 2026-05-28 + Spool delta)
- `stamp_ecu_swap` MUST NOT expose UPDATE on identity columns. Attempting `--update-existing-signature` (or similar update flag) is a CLI-level raise: "stamp_ecu_swap does not support updates to identity columns; close + open a new row via swap-event semantics instead". Even a typo-correction needs to go through close+open + audit trail (CIO ratifies acceptable cost).
- NEW: `add_ecu_note` CLI lands in this Story (folded; small enough). `python -m server.cli.add_ecu_note --vehicle-info-id <id> --text "..."` appends a timestamped line to `vehicle_info.notes`. Subsequent calls append; never overwrite. Empty `notes` allowed (NULL preserves "no notes ever recorded" semantic).
- Idempotency on `stamp_ecu_swap` (same signature + matching install timestamp) = no-op + log message; double-protection against accidental row duplication.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- CLI `python -m server.cli.stamp_ecu_swap --signature "<id>" --cal-signature "<id-or-null>" --as-of "YYYY-MM-DD[THH:MM:SSZ]"` records a swap event: sets `ecu_removal_timestamp_utc` on the currently-active row to the `--as-of` timestamp + inserts a new row with the new signature + NULL removal timestamp.
- CLI `python -m server.cli.show_ecu_lineage` lists all historical ECU stamps with install/removal timestamps + signatures (output: text table, sortable by install timestamp; current row highlighted).
- **Idempotency: re-running `stamp_ecu_swap` with the same signature + as-of timestamp is a no-op** (PM choice per Atlas Refinements row 8 -- preferred over fail-loudly; cleaner UX). Implementation: detect existing row with matching `(ecu_signature, ecu_install_timestamp_utc)`; if found + currently-active, exit 0 with INFO log "ECU swap already stamped; no-op". If found + closed, exit non-zero with ERROR.
- **Identity-immutability CLI raise wired**: `stamp_ecu_swap` does not expose UPDATE on identity columns; attempting `--update-existing-signature` (or equivalent update flag) raises with documented error message pointing to close+open semantics.
- **`add_ecu_note` CLI subcommand wired with tests**: `python -m server.cli.add_ecu_note --vehicle-info-id <id> --text "..."` appends timestamped lines to `vehicle_info.notes`; subsequent calls append (never overwrite); invalid `--vehicle-info-id` raises clean.
- Atomicity: stamp_ecu_swap close + open executed in a single DB transaction; partial-failure rollback preserves the invariant (exactly one currently-active row at all times).
- Unit tests cover: clean swap; idempotent re-run; attempt to stamp before US-365 migration ran (graceful error); attempt to stamp when no currently-active row exists (graceful error -- bootstrap scenario for US-367 backfill needs different code path); `add_ecu_note` happy path + invalid FK error.
- Both CLIs documented in module docstrings + brief usage note in `offices/pm/` or `docs/` (Spool + CIO need to invoke `stamp_ecu_swap` + `add_ecu_note` going forward).

## Conditional outcomes
- if `stamp_ecu_swap` re-runs with same `--signature` + `--as-of`, then exit 0 with INFO "no-op" (idempotent).
- if `stamp_ecu_swap` runs with same `--signature` but DIFFERENT `--as-of`, then exit non-zero with ERROR "signature already stamped at different timestamp; deliberate stamp required" (prevents silent timestamp rewrite).
- if `show_ecu_lineage` runs against empty `vehicle_info`, then exit 0 with informational "no ECU lineage recorded yet" (graceful empty-state).
- if Spool needs prefix-namespace on notes (e.g., `[KNOCK]`, `[CAL_DRIFT]`), that ergonomics lands as separate Story; out of scope here.
- if `stamp_ecu_swap` is run via direct Alembic-style ad-hoc SQL (CLI bypass), then no enforcement; documented in table comment as anti-pattern.

# US-379 — Fix stale harness fixture (`drive_id` → `summary_id`) in deploy-context simulator

## Context

The US-371 column rename (`drive_statistics.drive_id` → `summary_id`) landed
coherently in the production ORM + analytics + migration (verified live: the
**drive-27 IRL drill wrote `drive_statistics` — 16 params / 4771 samples**, ORM
and prod schema agree). The Sprint 43 integration (`72172a2`) renamed the ORM
calls + assertions inside
`tests/integration/test_deploy_context_drive_simulator.py::TestHarnessIntegrity::test_harnessTooling_canCatchSchemaVsOrmDivergence_synthetic`
(5 swaps) **but missed the hand-written "pre-v0009 historical" `CREATE TABLE
drive_statistics (...)` block** (lines ~783-797), which still declares the table
with the old `drive_id` column + `PRIMARY KEY (drive_id, parameter_name)` + FK on
`drive_id`.

Result: the production ORM INSERT now emits `summary_id`, the synthetic table
only has `drive_id`, and the test raises
`OperationalError: table drive_statistics has no column named summary_id`
**before** it reaches its intended `data_quality`-missing assertion (the I-041
retroactive proof). The test trips RED on `tests/integration`.

This is a **test-fixture maintenance miss, not a product defect.** It rode latent
through both the V0.28.1 and V0.28.2 deploys because the deploy/validation gate
runs `pytest tests/server -m "not slow"` (1081 passed) and does not include
`tests/integration`. Discovered during the V0.28 `/chain-validated` full-suite
pass on 2026-06-05; CIO ratified "merge now + file fast-follow."

## Goal

As the test suite, the synthetic divergence fixture should declare its historical
`drive_statistics` table with the **current** non-`data_quality` columns
(`summary_id` PK + FK to `drive_summary(id)`), so the test once again provokes the
intended `data_quality`-missing `OperationalError` and proves the harness catches
the I-041 schema-vs-ORM seam — rather than failing on a `summary_id`-missing
artifact of the rename.

## Definition of Done

- The hand-written `CREATE TABLE drive_statistics (...)` block in
  `test_harnessTooling_canCatchSchemaVsOrmDivergence_synthetic` uses `summary_id`
  (NOT NULL) as the column, `PRIMARY KEY (summary_id, parameter_name)`, and
  `FOREIGN KEY (summary_id) REFERENCES drive_summary(id)`.
- No production `src/` change (this is test-only; product already correct).
- The test passes for the right reason: Phase 1 raises `OperationalError`
  mentioning `data_quality`; Phase 2 (re-ADD `data_quality`) INSERT succeeds.

## Validation Criteria

- (run `pytest tests/integration/test_deploy_context_drive_simulator.py::TestHarnessIntegrity::test_harnessTooling_canCatchSchemaVsOrmDivergence_synthetic -v`)
  → (PASSED)
- (run `pytest tests/integration -m "not slow" -q`) → (no `summary_id`-missing failures introduced by this change)
- (grep the test for the historical CREATE TABLE block) → (declares `summary_id`, not `drive_id`)

## Conditional Outcomes

- If the historical-shape table is intentionally meant to predate the rename
  (i.e., the test wants to prove the harness catches BOTH the column-rename AND
  the `data_quality` gaps): split into two assertions and document which symptom
  each phase proves — but keep the default single-symptom (`data_quality`) intent
  unless Atlas/Argus rule otherwise.

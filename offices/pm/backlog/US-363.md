# US-363 — Server tripwire: `data_quality='attribution_anomaly'` integration

## Goal
As a server-side compute path producing `drive_summary` + `drive_statistics` rows, I want to invoke `detect_overlapping_drives` and stamp `data_quality='attribution_anomaly'` on rows whose drive_id has any overlap so that the V0.27.18 dual-attribution pattern (or any recurrence) is observable downstream as a per-row flag rather than silently produced.

The tripwire is **observability not refusal**: rows with `data_quality='attribution_anomaly'` are still readable by all downstream consumers; reports + CLIs degrade gracefully (flag the row in output; never 500 / never drop the row).

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Alembic v0010 substep extends `data_quality` enum / CHECK constraint to allow value `'attribution_anomaly'` on both `drive_summary` + `drive_statistics`.
- `compute_drive_summary(drive_id)` invokes `detect_overlapping_drives(session, drive_id)`; non-empty result → writes `data_quality='attribution_anomaly'`; empty result → writes `'full'` (or pre-existing value if richer).
- Same wiring applied to `compute_drive_statistics(drive_id)` (or equivalent server-side analytics writer per B-104 Step 1 shape).
- Downstream consumers verified: reports/CLIs that read `drive_summary.data_quality` surface the flag visibly (e.g., `[attribution_anomaly]` marker in CLI output) but do not 500, drop the row, or refuse to render the rest of the report.
- Unit tests cover: drive_id with overlap → row written with `'attribution_anomaly'`; drive_id without overlap → row written with `'full'`; downstream CLI consumer reads anomaly row + renders gracefully.
- Idempotent: re-running compute on same drive_id produces identical row (no diff on re-run; per US-350 idempotency pattern).
- Atlas Rule 10 sign-off recorded (load-bearing schema change -- enum extension on attributed authoritative table).

## Conditional outcomes
- if any downstream consumer (report.py / CLI / Ollama prompt template / analytics export) hard-fails on the new enum value, then patch the consumer before claiming completion (graceful degradation = DoD per Atlas Refinements row 5).
- if a consumer chooses to FILTER anomaly rows out by default, then it MUST log a one-line WARNING per anomaly row encountered so the omission is visible in operator logs (not silent suppression).

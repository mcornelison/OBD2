# US-370 — `speed_pid_calibration` table + 2-ECU seed; FK to `vehicle_info.ecu_signature`

## Goal
As a server-side analytics consumer aggregating SPEED PID for distance / gear-inference, I want a `speed_pid_calibration` table keyed by `ecu_signature` (FK to `vehicle_info`) carrying a multiplicative `correction_factor` so that the new modified-EPROM ECU's ~2× SPEED-reading drift gets corrected per-drive without polluting analytics.

**NOTE Q2 pending**: SPEED-PID new-ECU seed `correction_factor` initial value = **0.5** (rough estimate from Drive 26 telemetry; Spool empirical refinement Q2 PENDING; may be revised to a GPS-correlation value before sprint freeze or in a V0.28.1 follow-up).

## Scope refinement (Spool Q2 + Atlas 2026-05-28 ACK -- provenance column)
Add `provenance TEXT NOT NULL` column on `speed_pid_calibration` (per Spool Q2 + Atlas ACK 2026-05-28). Provenance values from Atlas + Spool: `rough-seed-drive-26-gear-math`, `empirical-gps-correlation`, `identity-1.0-oem`, `vendor-spec`, `default`. Open vocabulary; analytics consumers gate on prefix.

Seed rows:
- Prior ECU: `correction_factor=1.0, provenance='gear-math-drive-18-3rd-gear-fit'`
- New ECU: `correction_factor=0.5, provenance='rough-seed-drive-26-gear-math'`

Spool's empirical refinement post-sprint via GPS-correlation drive updates the row to `provenance='empirical-gps-correlation'` (new-row close+open pattern OR direct UPDATE -- Spool's call; document in `notes` column on calibration table if useful).

# US-371 — Column rename: `drive_statistics.drive_id` → `summary_id` + consumer updates

## Goal
As a server-side analytics consumer reading `drive_statistics`, I want the misleading `drive_id` column renamed to `summary_id` (which is what the column actually holds -- a `drive_summary.id` FK, NOT a Pi-assigned drive_id) so that future code authors aren't lied to about the column's meaning.

The rename is **complete, not additive**: no deprecated `drive_id` alias is retained. Spool's 2026-05-22 flag named this column as a semantic lie; an alias would perpetuate the lie.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Alembic v0010 substep: ALTER TABLE `drive_statistics` RENAME COLUMN `drive_id` TO `summary_id` (server-side; Pi-side `drive_statistics` was retired in US-351 so no Pi-side rename).
- ORM model updated: `src/server/models/drive_statistics.py` (or equivalent path) references `summary_id` not `drive_id`.
- All consumers grepped + updated (exhaustive enumeration per Atlas Refinements row 12):
  - `src/server/analytics/drive_statistics_compute.py`
  - `src/server/cli/recompute_drive_analytics.py`
  - `src/server/reports/*.py`
  - `src/server/services/analysis.py`
  - `tests/server/`
  - Spool's ad-hoc analytical SQL referencing `drive_statistics.drive_id` literally (search `offices/tuner/` for SQL fragments; route findings to Spool)
- Post-rename, `SELECT summary_id FROM drive_statistics LIMIT 1` succeeds AND `SELECT drive_id FROM drive_statistics LIMIT 1` FAILS (rename is complete; no alias).
- Existing analytics queries return the same results post-rename (semantic-equivalent rewrites; not new behavior).
- Atlas Rule 10 sign-off recorded (load-bearing schema rename).

## Conditional outcomes
- if a grep surfaces a consumer that references `drive_statistics.drive_id` literally + the consumer is outside `src/server/` (e.g., in `offices/tuner/` scripts or `docs/`), then route an A2AL note to the owning agent (Spool or PM) + update the reference; do not silently leave it broken.
- if MariaDB version on chi-srv-01 doesn't support RENAME COLUMN syntax cleanly, then fallback to ADD COLUMN summary_id + UPDATE + DROP COLUMN drive_id in the same migration substep (still no alias).

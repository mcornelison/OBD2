# US-368 — `dtc_freeze_frame` table + Pi-side Mode 02 capture on MIL_ON

## Goal
As a tuner doing post-mortem on a DTC trigger event, I want the engine state at the moment the DTC tripped (Mode 02 freeze-frame: 16 PIDs mirroring Mode 01) captured + stored alongside the DTC log entry so that forensic analysis has "what was the engine doing when this error tripped" context.

Per Q4 ruling: `dtc_freeze_frame` carries an FK to `vehicle_info.id` (specific row, not "currently active"); forensic immutability comes from `vehicle_info`'s **append-only invariant** (corrections = close prior row + open new; never UPDATE the row a freeze-frame points to).

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Alembic v0010 substep creates `dtc_freeze_frame` table with shape:
  - `id INT PRIMARY KEY AUTO_INCREMENT`
  - `dtc_log_id INT NOT NULL` FK -> `dtc_log.id`
  - `captured_at_timestamp_utc TIMESTAMP NOT NULL`
  - `pid_responses_json JSON NOT NULL` (16-PID dictionary)
  - `vehicle_info_id INT NOT NULL` FK -> `vehicle_info.id` (per Q4 ruling -- specific row, not denormalized text)
- `vehicle_info` table comment / docstring documents **append-only invariant**: corrections via close-prior-row + open-new-row; never UPDATE a row that a `dtc_freeze_frame` points to.
- Append-only invariant enforced via writer-path discipline: no UPDATE statements on `vehicle_info` rows in `src/server/`; corrections go through `stamp_ecu_swap` (or an equivalent close+open helper).
- Regression test: vehicle_info row mutation attempt after a `dtc_freeze_frame` points to it FAILS (or, if pure-writer-path enforcement, the test asserts no code path UPDATEs vehicle_info rows -- grep-based or runtime-assertion).
- Pi-side: MIL_ON event detection (existing wiring per F-109 description) triggers Mode 02 enumeration of all 16 supported PIDs; writes single row keyed by `(dtc_log_id, captured_at_timestamp_utc)` with `vehicle_info_id` resolved to the currently-active row at capture time.
- Pi-side unit test: synthetic MIL_ON event triggers Mode 02 capture path; `dtc_freeze_frame` row written with 16-PID JSON + FK to currently-active vehicle_info row.
- Atlas Rule 10 sign-off recorded (load-bearing schema + Pi DTC detection path touched).

## Conditional outcomes
- if Spool later vetoes the append-only invariant (CIO veto path open per Q4 -- vehicle_info append-only feels too restrictive for typo corrections), then the implementation pivots to FK + denormalized `ecu_signature_at_capture` TEXT column; document as separate Story for V0.28.1.
- if the OBD simulator under `simulate` mode doesn't return Mode 02 responses, then add Mode 02 PID enumeration to the simulator (in-scope for this Story; the Pi unit test depends on the simulator).
- if a DTC trips but Mode 02 PIDs are unavailable (some real-world hardware paths drop the freeze-frame), then write the row with `pid_responses_json = {}` + a `notes` field explaining the gap (graceful degradation).

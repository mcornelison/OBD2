## Description

Implement tuning-driven alert thresholds for all Phase 1 (OBD-II only, stock ECU) parameters. The alert system module already exists (US-017 complete), but it uses placeholder or generic values. This item wires in Spool's exact, DSM-community-grounded threshold values for 6 parameters.

Every alert has three levels: **Normal** (green), **Caution** (yellow), **Danger** (red). Some parameters have a fourth informational level (e.g., "cold" for coolant). These are the values the driver sees on the 3.5" display and the values that trigger audible alerts.

**Source of truth**: All thresholds come from Spool (Tuning SME), grounded in 4G63 community knowledge and DSMTuners.com data. PM Rule 7 applies — no fabricated data.

## User Stories

### US-107: Implement Coolant Temperature Alert Thresholds

**As a** driver, **I want** the system to alert me at specific coolant temperature ranges **so that** I can prevent head gasket failure.

**Acceptance Criteria:**
- [ ] Coolant temp thresholds implemented with 4 levels:
  - **Cold**: < 160F — Blue indicator (info only). Engine not at operating temp. Normal during warmup.
  - **Normal**: 180-210F — Green. No action.
  - **Caution**: 210-220F — Yellow, audible beep. Alert driver, monitor closely, reduce load.
  - **Danger**: > 220F — Red, continuous alert. **PULL OVER. Engine off. Head gasket risk.**
- [ ] Thresholds sourced from config (not hardcoded) so they can be adjusted
- [ ] Alert log entry created at Caution and Danger transitions with severity and message
- [ ] Tests validate each threshold boundary (159F=cold, 160F=cold, 179F=cold, 180F=normal, 210F=normal, 211F=caution, 220F=caution, 221F=danger)

**Rationale (from Spool):** The 4G63 MLS head gasket loses clamp force when head bolts stretch from heat. Above 220F, the gasket weeps coolant into #4 cylinder. This is the most common 4G63 failure mode and it's entirely preventable with temperature monitoring.

**Developer Reference — Example Alert Escalation (from Spool):**
```
timestamp,           coolant_temp_f, alert_level
2026-07-15 14:00:00, 165,           cold       (blue, info)
2026-07-15 14:05:00, 182,           normal     (green)
2026-07-15 14:10:00, 192,           normal     (green)
2026-07-15 14:30:00, 195,           normal     (green)
2026-07-15 14:45:00, 208,           normal     (green)
2026-07-15 14:50:00, 212,           caution    (yellow — alert triggered)
2026-07-15 14:52:00, 218,           caution    (yellow — still in caution)
2026-07-15 14:53:00, 221,           danger     (red — DANGER alert triggered)
```

**Expected system behavior:**
- At 14:00 — Display shows blue coolant indicator (cold engine, normal during warmup)
- At 14:05 — Display transitions to green (engine warming up, crossed 180F)
- At 14:50 — Display turns yellow, audible beep, alert logged: "Coolant temp 212F — monitor closely"
- At 14:53 — Display turns red, continuous audible alert, alert logged: "DANGER: Coolant temp 221F — pull over immediately, head gasket risk"

**Expected alert log entries:**
```
id, timestamp,            parameter,    value, severity, message
1,  2026-07-15 14:50:00, COOLANT_TEMP, 212,   CAUTION,  "Coolant temperature elevated (212F). Monitor closely."
2,  2026-07-15 14:53:00, COOLANT_TEMP, 221,   DANGER,   "DANGER: Coolant temperature critical (221F). Risk of head gasket failure. Reduce load immediately."
```

### US-108: Implement Short-Term Fuel Trim (STFT) Alert Thresholds

**As a** driver, **I want** the system to alert me when STFT values are abnormal **so that** I can detect fueling problems before they damage the engine.

**Acceptance Criteria:**
- [ ] STFT thresholds implemented with 3 levels:
  - **Normal**: -5% to +5% — ECU making small corrections. Healthy.
  - **Caution**: -10% to -5% OR +5% to +10% — ECU working harder to correct. Something may be developing. Log for trend analysis.
  - **Danger**: < -15% OR > +15% — ECU at correction limit. Active fueling problem. Alert driver. **If positive (lean): could be vacuum leak, weak fuel pump, or clogged injector.**
- [ ] Thresholds sourced from config
- [ ] Alert messages distinguish positive (lean) from negative (rich) deviations
- [ ] Tests validate symmetric threshold boundaries (-15.1=danger, -15=danger, -10=caution, -5=caution, -4.9=normal, +4.9=normal, +5=caution, +10=caution, +15=danger, +15.1=danger)

**Rationale (from Spool):** STFT shows what the ECU is doing RIGHT NOW to correct fueling. Large positive STFT means the engine is running lean and the ECU is adding fuel to compensate. On a turbocharged engine, lean = detonation = dead engine.

### US-109: Implement Timing Advance Alert Thresholds

**As a** driver, **I want** the system to detect timing retard events **so that** I can identify possible knock/detonation.

**Acceptance Criteria:**
- [ ] Timing advance thresholds implemented with 3 levels:
  - **Normal**: Matches baseline for RPM/load point. ECU is happy with combustion.
  - **Caution**: Drops > 5 degrees below baseline suddenly. Possible knock event, ECU pulling timing. Log it. If repeated at same RPM/load, flag for review.
  - **Danger**: Goes to 0 or negative under load. Active detonation. ECU is pulling all timing. **Reduce throttle immediately. Do not continue at this load.**
- [ ] System must track a per-RPM/load baseline (established from initial drives)
- [ ] Sudden drops are detected relative to baseline, not absolute values
- [ ] Repeated timing pulls at the same RPM/load point are flagged as a pattern
- [ ] Tests validate baseline comparison logic and sudden-drop detection

**Rationale (from Spool):** Without ECMLink, timing advance is our only window into knock activity. The stock ECU will retard timing when the knock sensor detects detonation. A sudden drop in timing at a specific RPM/load point means the engine is knocking there.

### US-110: Implement Engine RPM Alert Thresholds

**As a** driver, **I want** RPM boundary alerts **so that** I can avoid over-revving and detect idle issues.

**Acceptance Criteria:**
- [ ] RPM thresholds implemented with 4 levels:
  - **Low idle**: < 600 RPM — Idle too low. Possible vacuum leak or IAC issue.
  - **Normal**: 600-6500 RPM — No action.
  - **Caution**: 6501-7000 RPM — High RPM warning. Stock redline approaching (factory redline = 7000 RPM on 97-99 2G).
  - **Danger**: > 7000 RPM — **Over-rev. Valve float risk on stock springs. Factory redline exceeded.**
- [ ] Thresholds sourced from config
- [ ] Tests validate all boundaries (599=low_idle, 600=normal, 6500=normal, 6501=caution, 7000=caution, 7001=danger)

**Rationale (from Spool):** The 97-99 2G Eclipse GST has a factory redline of 7000 RPM (softer cam than 95-96). Stock valve springs are not rated for sustained operation above 7000 RPM — valve float risk increases rapidly above redline. The earlier draft of this spec listed 7200 as the danger threshold, which was incorrect. Corrected 2026-04-12 by Spool.

### US-111: Implement Battery Voltage Alert Thresholds

**As a** driver, **I want** charging system alerts **so that** I can detect alternator or regulator failure.

**Acceptance Criteria:**
- [ ] Battery voltage thresholds implemented with 3 levels (engine running):
  - **Normal**: 13.5-14.5V — Charging system healthy.
  - **Caution**: 12.5-13.5V OR 14.5-14.8V — Low = weak alternator. High = regulator starting to fail.
  - **Danger**: < 12.0V OR > 15.0V — **Low = charging failure, engine may stall. High = regulator failed, will cook battery and electronics.**
- [ ] PID source: 0x42 (Control Module Voltage) or hardware UPS reading
- [ ] Thresholds sourced from config
- [ ] Alert messages distinguish low-voltage from high-voltage conditions
- [ ] Tests validate both low and high boundary conditions

### US-112: Implement Intake Air Temperature (IAT) Alert Thresholds

**As a** driver, **I want** IAT alerts **so that** I can detect heat soak and knock risk conditions.

**Acceptance Criteria:**
- [ ] IAT thresholds implemented with 4 levels:
  - **Normal**: Ambient to 130F — Normal for turbocharged intake.
  - **Caution**: 131-160F — Heat soak building. Power loss. Increased knock risk. Stock side-mount intercooler heat soaks quickly under sustained boost.
  - **Danger**: > 160F — **Significant knock risk at boost. Reduce load. Consider FMIC upgrade.**
  - **Sensor Failure**: Fixed at -40F (-40C) — IAT sensor disconnected or failed. **This is a known 2G quirk — the IAT sensor is inside the MAF housing on 2G DSMs. Constant -40 = sensor disconnected or failed.**
- [ ] Sensor failure detection: if IAT reads exactly -40F continuously, flag as sensor failure (not cold air)
- [ ] Thresholds sourced from config
- [ ] Tests validate boundaries (130F=normal, 131F=caution, 160F=caution, 161F=danger) and sensor failure detection (-40F held for N consecutive readings = failure)

**Rationale (from Spool):** The earlier draft of this spec had a gap (150-160F undefined). Corrected 2026-04-12 by Spool. Caution range now covers 131-160F continuously — all of this range represents progressive heat soak with progressive knock risk, so they share the same alert level.

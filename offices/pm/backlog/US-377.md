# US-377 — `data_quality` column too narrow for `'attribution_anomaly'` (VARCHAR(16) < 19 chars) — chain-blocking

## Discovery
Surfaced during the **V0.28.1 IRL drill** (2026-06-01) while executing US-364
(`recompute_drive_analytics --drive-id-range 23-25` against chi-srv-01 production
`obd2db`). The recompute **correctly detected** drives 23 + 24 as attribution
anomalies and attempted to stamp them, but the write failed:

```
sqlalchemy.exc.DataError: (pymysql.err.OperationalError) (1406,
  "Data too long for column 'data_quality' at row 1")
UPDATE drive_summary SET data_quality='attribution_anomaly' WHERE id=31
```

Result: `success=1 (drive 25, 'full'), failed=2 (drives 23+24), attribution_anomalies=0`.
No data corruption — the failed UPDATEs rolled back transactionally; drives
23/24/25 remain `data_quality='full'` with intact start/end/row_count.

## Root cause
`'attribution_anomaly'` is **19 characters**. Both columns are **`VARCHAR(16)`**:
- `drive_statistics.data_quality` — `VARCHAR(16)` added by **US-357 / v0009** (original enum `full`/`sparse`/`below_threshold` all fit ≤14); **US-363 / v0010 widened the CHECK** to add `attribution_anomaly` but **did NOT widen the column**.
- `drive_summary.data_quality` — `VARCHAR(16)` created by **US-363 / v0010** with the CHECK already listing `attribution_anomaly` from the start.

The CHECK constraints (`ck_drive_summary_data_quality`, `ck_drive_statistics_data_quality`)
explicitly permit a value the column width physically rejects — a self-contradictory
schema. ORM origin: `src/server/db/models.py:991` (`data_quality: Mapped[str] =
mapped_column(String(16), ...)`) and the `drive_statistics` declaration.

## Why tests missed it (false-pass class)
Unit/migration tests run on SQLite, which does **not** enforce `VARCHAR` length;
MariaDB does. This is the same SQLite-vs-MariaDB enforcement gap that produced the
historical false-pass cluster (V0.27.7/.16/.17). The 1058 green server tests did
not catch it because no test inserts `'attribution_anomaly'` against MariaDB.

## Goal
As an operator running the attribution-anomaly tripwire backfill, I want the
`data_quality` columns wide enough to hold every value their CHECK permits, so that
`recompute_drive_analytics` can stamp `'attribution_anomaly'` on drives 23+24 and
US-364 (and the F-005/F-007 HOLD release) can complete.

## Definition of Done (at least 1 required)
- ORM: both `data_quality` columns widened from `String(16)` to **`String(20)`** (headroom over the 19-char value) in `src/server/db/models.py`.
- New forward-only migration **v0012**: `ALTER TABLE drive_summary MODIFY data_quality VARCHAR(20) NOT NULL DEFAULT 'full'` + same for `drive_statistics`; CHECK + index + default preserved; idempotent (INFORMATION_SCHEMA column-length probe).
- A test that inserts/updates `'attribution_anomaly'` against the **real MariaDB enforcement semantics** (or asserts the column length ≥ max CHECK-permitted value length) so this class can't regress.
- After deploy + migration: re-running `recompute_drive_analytics --drive-id-range 23-25` stamps drives 23+24 `attribution_anomaly`, drive 25 stays `full` (unblocks US-364).

## Conditional outcomes
- if any OTHER column constrained by a CHECK enum has the same width-vs-CHECK mismatch (audit `dtc_log.status`, `dtc_code`, etc.), then fold the audit findings into this Story rather than shipping a partial fix.

# US-369 — Server sync + `show_dtc_freeze_frame` CLI

## Goal
As an operator wanting to inspect a freeze-frame after a DTC trigger event, I want the Pi-to-server sync to deliver `dtc_freeze_frame` rows + a `show_dtc_freeze_frame --dtc-log-id N` CLI that prints the 16-PID dictionary + joins through to the correct `vehicle_info` row by FK so that forensic context is available without raw-SQL excursions.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Pi-to-server sync (`src/pi/sync/` or equivalent) maps Pi-side `dtc_freeze_frame` rows to server-side rows; round-trip preserves all 16 PIDs + FKs.
- `python -m server.cli.show_dtc_freeze_frame --dtc-log-id N` prints:
  - The `dtc_log` row (DTC code + timestamp).
  - The `dtc_freeze_frame` row (captured_at + 16-PID dictionary).
  - The `vehicle_info` row joined via FK (`ecu_signature`, `cal_signature`, install/removal timestamps).
- Q4 round-trip criterion: `show_dtc_freeze_frame --dtc-log-id N` correctly joins through to the vehicle_info row that was active at capture time, even if `vehicle_info` has had subsequent rows added via `stamp_ecu_swap`.
- Sync delivery test: synthetic Pi-side `dtc_freeze_frame` row syncs to server; server-side `show_dtc_freeze_frame` returns the synced row with all 16 PIDs intact.
- CLI handles missing freeze-frame gracefully: `show_dtc_freeze_frame --dtc-log-id N` against a dtc_log row that has no freeze-frame exits 0 with informational "no freeze-frame recorded for this DTC".

## Conditional outcomes
- if a sync delivers a `dtc_freeze_frame` row whose `vehicle_info_id` FK doesn't exist server-side (cross-tier ID drift), then sync FAILS LOUDLY with the cross-tier-ID issue logged; do not silently re-resolve.
- if `pid_responses_json` is empty (graceful degradation path from US-368 V-6), then `show_dtc_freeze_frame` displays "freeze-frame captured but Mode 02 PIDs unavailable" + notes field.
- if multiple freeze-frame rows exist for one `dtc_log_id` (shouldn't happen by design but defensive), then `show_dtc_freeze_frame` displays the most recent by `captured_at_timestamp_utc` + WARN about the duplicate.

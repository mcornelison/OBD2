## Description

Spool audited all delivered sprint 1 and sprint 2 code for tuning-domain values and found **7 variances** between the code and his corrected tuning specifications. 5 are critical (actively wrong runtime values), 2 are minor. This backlog item bundles all 5 fixes as a single cleanup effort.

**Background**: The codebase has two parallel threshold systems:
1. **Tiered threshold system** (`obd_config.json` → `tieredThresholds`, `src/alert/tiered_thresholds.py`) — new, correct after sprint 4 hotfix
2. **Legacy profile threshold system** (`obd_config.json` → `profiles.availableProfiles[*].alertThresholds`, `src/alert/thresholds.py`) — older, contains stale/wrong values that never got updated when the tiered system was added

Spool's audit focuses on the legacy system's wrong values. Whether to ultimately deprecate the legacy system is an architecture decision for a future sprint — this backlog item just corrects the values so the legacy system is at least accurate.

**Reference**: `offices/pm/inbox/2026-04-12-from-spool-code-audit-variances.md`

## User Stories

### US-140: Fix Legacy Profile Coolant Critical Threshold

**Problem**: The value `coolantTempCritical: 110` appears in 6 files with no unit specified. 110°F is cold-engine territory (engines run 180-210°F normally) — this alert would fire constantly. If meant as °C (110°C = 230°F), that's in range but the unit isn't explicit.

**Acceptance Criteria:**
- [x] Update all 6 locations of `coolantTempCritical: 110` (or 115 in performance profile) to `220`:
  - `src/alert/thresholds.py:139`
  - `src/alert/manager.py:72`
  - `src/profile/types.py:45`
  - `src/obd_config.json:108` (daily profile)
  - `src/obd_config.json:119` (performance profile, currently 115)
  - `src/obd/obd_config_loader.py:350`
  - `src/obd/config/loader.py:350`
- [x] Add explicit `unit: "fahrenheit"` field to profile threshold schema where applicable
- [x] Update any tests asserting the old values (grep for `110` and `115` in `tests/` near coolant context)
- [x] Full test suite passes (pytest tests/)
- [x] Config JSON validates (validate_config.py)

**Rationale (from Spool)**: The legacy profile threshold system was built before the tiered system. When the tiered system was added with correct values (180/210/220), the legacy values weren't updated. 220°F matches Spool's head-gasket danger threshold.

### US-141: Complete RPM Hotfix in Legacy Profile System

**Problem**: US-139 (sprint 4 hotfix) updated `tieredThresholds.rpm.dangerMin` from 7200 → 7000. It did **not** update the legacy profile system. Performance profile still has `rpmRedline: 7200`.

**Acceptance Criteria:**
- [x] Update `src/obd_config.json:124`: performance profile `rpmRedline` from `7200` to `7000`
- [x] No test fixtures reference `rpmRedline: 7200` (all use independent values)
- [x] Full test suite passes (1517 passed, 1 skipped)
- [x] Config JSON validates (`python -m json.tool` + `validate_config.py`)

**Rationale (from Spool)**: 97-99 2G factory redline is 7000 RPM. US-139 fixed the tiered system; this completes the fix in the legacy profile system.

### US-142: Correct Stock Turbo Boost Pressure Limits

**Problem**: Legacy profile has `boostPressureMax: 18` psi. The CIO's stock TD04-13G turbo caps safely at 14-15 psi. 18 psi is above the turbo's efficiency range and into compressor surge territory.

**Acceptance Criteria:**
- [x] Update `src/obd_config.json:127`: `boostPressureMax` from `18` to `15`
- [x] Update `src/alert/thresholds.py:140`: matching default from `18` to `15`
- [x] Add code comment near both values: "Stock TD04-13G turbo safe ceiling. Update when turbo is upgraded (16G/20G)." (added to thresholds.py; JSON does not support comments)
- [x] Update any tests asserting the old 18 psi value (none found — test fixtures use independent values)
- [x] Full test suite passes (1517 pass)

**Rationale (from Spool)**: Stock turbo is unsafe above 15 psi. Setting the alert at 18 means boost is already in danger territory before the alert fires. 15 psi is the correct ceiling for this vehicle in its current state.

**Rationale (from Spool)**: Display stubs aren't runtime-active yet (Phase 2 blocked on ECMLink) but a developer implementing Phase 2 would see these and assume they're reasonable ballpark values. They're dangerous.

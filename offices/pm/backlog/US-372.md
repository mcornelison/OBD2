# US-372 — `drive_summary.drive_id` NULL decision + implementation

## Goal
As a server-side analytics consumer reading `drive_summary`, I want the `drive_id` NULL smell (post-B-104 Step 1 server-compute rows leave `drive_id` NULL because compute paths use `summary_id` to join `drive_statistics`) resolved via **Q1 option (a) Backfill + invariant** so that `drive_summary.drive_id` is a semantically meaningful column going forward + future authors aren't bitten by the NULL.

Per Q1 ruling (CIO + Atlas 2026-05-28): option (a) Backfill + invariant. Option (b) SSOT-purist drop deferred to V0.28+ B-076 broader-normalization sprint where consumer-grep can be done thoroughly.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Alembic v0010 substep performs in order:
  - (i) `UPDATE drive_summary SET drive_id = source_id WHERE drive_id IS NULL AND source_id IS NOT NULL` -- clears NULL drive_id on rows that have a non-NULL source_id.
  - (ii) After UPDATE: `ALTER TABLE drive_summary ADD CONSTRAINT chk_drive_id_source_id CHECK ((drive_id IS NULL AND source_id IS NULL) OR (drive_id = source_id))` -- enforces the invariant going forward.
- Writer-path discipline: both server-side `src/server/analytics/drive_summary_compute.py` AND Pi-side `src/pi/.../drive_summary.py` (current paths) set BOTH `drive_id` AND `source_id` columns on INSERT/UPSERT -- never one-without-the-other when the value is known.
- Regression test asserts invariant under both write paths:
  - Server-side compute INSERT -> both columns set + equal.
  - Pi-side write INSERT -> both columns set + equal.
  - Direct INSERT violating the CHECK constraint FAILS.
- Substep ordering documented in v0010 migration docstring: UPDATE BEFORE ALTER (per Atlas Migration Risk note Refinements row 16).
- Atlas Rule 10 sign-off recorded (load-bearing schema constraint).

## Conditional outcomes
- if any pre-existing `drive_summary` row has `drive_id IS NOT NULL AND source_id IS NULL` (asymmetric case), then the invariant is violated by pre-existing data; the migration must UPDATE such rows to `source_id = drive_id` BEFORE adding the CHECK constraint -- OR the case must be enumerated + manually dispositioned. Ralph audits pre-existing data + documents disposition.
- if a writer-path code site only knows one of the two columns at write time (e.g., reading from a legacy payload that only has source_id), then write `(drive_id=source_id, source_id=source_id)` rather than `(drive_id=NULL, source_id=source_id)` so the invariant holds (drive_id is a duplicate of source_id per Q1; they're the same Pi-emitted id).

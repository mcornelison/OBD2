## Summary

Phase the Raspberry Pi from validated simulator through physical car installation to real-world data collection using a crawl/walk/run/sprint progression.

- **Crawl**: Deploy to Pi hardware, validate existing 164-file codebase on ARM, prove display and startup/shutdown
- **Walk**: Connect to Chi-Srv-01, push simulated data through real sync pipeline

- **Sprint**: Real driving, full lifecycle, Spool data quality review, interactive touch display

## Paused Stories (pending external resolution)

### US-180 — Hardware subsystem (UPS + GPIO) — PAUSED Sprint 10 Session 20

**Paused**: 2026-04-17 — CIO directive
**Blocker**: X1209 UPS HAT hardware. `i2cdetect -y 1` returns empty despite power cycle, reseat, and pushbutton press. The HAT's own power LED fades on/off (weak) when USB-C feeds the HAT — a classic undervoltage symptom. May also explain the dead I2C (fuel-gauge chip unpowered through a sagging rail). Before rewriting ACs to assume no-telemetry, investigate:
- USB-C PSU wattage (X1209 wants 5.1V/5A; official Pi 5 PSU ideal)
- USB-C cable quality (thin / long / charge-only cables drop voltage at 5A)
- X1209 USB-C input stage fault (as a last resort)

If power rail stabilizes, re-probe — fuel gauge may come alive with no code change. If not, US-180 will be rescoped in a follow-up sprint (drop I2C telemetry ACs, keep software-only path already delivered by Ralph — 32 tests in `tests/pi/hardware/` cover TelemetryLogger rotation + GPIO button mock + UpsMonitor graceful degradation, all passing on Pi + Windows).

**Remains in sprint.json** with `status: blocked` for traceability; excluded from Sprint 10 progress counts.

Full history: `offices/pm/inbox/2026-04-17-from-rex-us180-x1209-no-i2c-presence.md` + commit `3448630`.

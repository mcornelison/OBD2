# US-365 — `vehicle_info` ECU + cal_signature columns + currently-active constraint

## Goal
As a server-side analytics consumer needing per-ECU calibration awareness (CIO swapped ECUs 2026-05-22), I want `vehicle_info` to carry `ecu_signature`, `cal_signature`, `ecu_install_timestamp_utc`, `ecu_removal_timestamp_utc` columns with a constraint enforcing exactly one currently-active row so that downstream analytics + CLIs can join per-drive analytics to the ECU that was active at the drive's time window.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- Alembic v0010 substep adds columns to `vehicle_info` table:
  - `ecu_signature TEXT NOT NULL`
  - `cal_signature TEXT NULL`
  - `ecu_install_timestamp_utc TIMESTAMP NOT NULL`
  - `ecu_removal_timestamp_utc TIMESTAMP NULL` (NULL = currently installed)
- Constraint enforces: exactly one row at any time has `ecu_removal_timestamp_utc = NULL` (the currently-active ECU). Implementation: partial unique index `(ecu_removal_timestamp_utc) WHERE ecu_removal_timestamp_utc IS NULL` OR equivalent CHECK + application-layer enforcement -- Ralph picks the MariaDB-supported shape.
- Migration path for pre-existing `vehicle_info` rows from prior VIN sync (which lack ECU columns): **backfill with prior-ECU signature** (mapped to US-367 backfill data; the prior ECU is well-known per CIO records + chain-merge history). The "leave NULL with pre-tracking semantic" alternative was rejected as ambiguous per Atlas Refinements row 7.
- ORM model (`src/server/models/vehicle_info.py` or equivalent) updated to reflect new shape.
- Atlas Rule 10 sign-off recorded (load-bearing schema change).

## Conditional outcomes
- if any pre-existing `vehicle_info` row predates 2026-05-22 swap, then backfill writes prior-ECU signature (per US-367 data) -- consistent with append-only invariant: the row's `ecu_removal_timestamp_utc` is set to the 2026-05-22 swap timestamp, NOT NULL.
- if Alembic detects more than one row with NULL `ecu_removal_timestamp_utc` post-backfill, then migration FAILS LOUDLY (invariant violated; do not silently choose a winner).
- if MariaDB version on chi-srv-01 doesn't support partial unique indexes, then implement via CHECK constraint + application-layer enforcement; document the choice in migration docstring.

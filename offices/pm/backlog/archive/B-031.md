## Description

Implement 6 server-side analysis definitions that run on Chi-Srv-01 when the Pi syncs a datalog. Each analysis has a specific input, specific output format, and a specific "so what" advisory that gets sent back to the Pi.

Also includes drive summary generation logic (used on both Pi and server).

## User Stories

### US-129: Implement Knock Correlation Analysis

**As a** driver reviewing server reports, **I want** knock events mapped to RPM/load cells **so that** I can adjust the tune at the specific cells where knock occurs.

**Purpose:** Identify WHERE the engine is knocking (at what RPM and load) so the tune can be adjusted.

**Acceptance Criteria:**
- [ ] **Input:**
  - All knock events from the drive (timestamp, knock_count, knock_sum)
  - Corresponding RPM, load, timing_advance, AFR, boost at each knock event timestamp
- [ ] **Process:**
  - Group knock events by RPM band (500 RPM bins: 2000-2500, 2500-3000, etc.) and load band (10% bins)
  - For each bin with knock: record average timing, average AFR, average boost at knock
  - Compare with knock-free bins at similar RPM/load
- [ ] **Output format** (exact):
  ```
  KNOCK CORRELATION REPORT — Drive {date} {time}
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Total knock events: {count}
  Total knock sum: {sum}
  
  Affected cells:
    RPM {low}-{high}, Load {low}-{high}%: {count} events, avg timing {deg} deg, avg AFR {afr}:1, boost {psi} psi
  
  Advisory: {specific recommendation — e.g., "Add fuel in the 4000-4500 RPM / 70-80% load cell."}
  ```
- [ ] Advisory distinguishes likely real knock from likely false knock:
  - Real knock: high load, boost present, lean AFR, significant timing pull
  - False knock: low load, vacuum, low RPM — probable heat shield rattle, exhaust tick, or road vibration
- [ ] Tests validate with Spool's worked example:
  ```
  Event 1: 4200 RPM, 75% load, 13.5 psi boost, 85% TPS, 11.8:1 AFR, timing 14→11 deg
    → LIKELY REAL KNOCK (high load + slightly lean AFR)
    → Action: Add fuel at 4000-4500 RPM / 70-80% load. Consider removing 1 degree timing.

  Event 2: 2100 RPM, 25% load, -15 inHg vacuum, 12% TPS, 14.7:1 AFR, timing 18→16 deg
    → LIKELY FALSE KNOCK (light cruise, no boost, low load)
    → Action: No tune change. If pattern repeats, check loose heat shields or exhaust hangers.
  ```
- [ ] Returns advisory text suitable for display on Pi

### US-130: Implement AFR Drift Trending Analysis

**As a** driver reviewing server reports, **I want** fuel trim trends across drives **so that** I can detect a developing lean or rich condition before it becomes dangerous.

**Purpose:** Detect gradual changes in air-fuel ratio that indicate a developing mechanical problem.

**Acceptance Criteria:**
- [ ] **Input:**
  - LTFT values from the last N drives (minimum 5 drives for meaningful trend)
  - Grouped by operating condition: idle, cruise, boost
- [ ] **Process:**
  - Calculate LTFT average per operating condition per drive
  - Plot trend over drives (store data points for display)
  - Flag if trend is moving consistently in one direction (> 0.5% per drive average)
- [ ] **Output format** (exact):
  ```
  AFR DRIFT REPORT — Last {N} Drives
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Idle LTFT trend: {trend}%/drive ({N} drives ago: {old}%, now: {new}%)
  Cruise LTFT trend: {trend}%/drive ({N} drives ago: {old}%, now: {new}%)
  Boost LTFT trend: {trend}%/drive ({N} drives ago: {old}%, now: {new}%)
  
  Advisory: {specific recommendation with possible causes in order of likelihood}
  ```
- [ ] Advisory includes ordered possible causes when trend is concerning:
  1. Fuel pump weakening (check fuel pressure with gauge — should be 43.5 psi base)
  2. Injector starting to clog (one or more)
  3. Small boost/vacuum leak developing
- [ ] Advisory includes action threshold: "If LTFT exceeds +8%, investigate before next WOT pull."
- [ ] Tests validate with Spool's worked example:
  ```
  Input (6 drives):
  drive_date,  ltft_idle_avg, ltft_cruise_avg, ltft_boost_avg
  2026-06-01,  +1.2%,         +0.8%,           +0.5%
  2026-06-08,  +1.5%,         +1.0%,           +1.2%
  2026-06-15,  +2.0%,         +1.1%,           +2.5%
  2026-06-22,  +2.8%,         +1.3%,           +3.8%
  2026-06-29,  +3.5%,         +1.5%,           +5.2%
  2026-07-06,  +4.2%,         +1.4%,           +6.8%

  Expected output:
  Idle LTFT: +0.6%/drive → MONITOR (approaching +5% investigation threshold)
  Cruise LTFT: +0.1%/drive → NORMAL (within expected variation)
  Boost LTFT: +1.3%/drive → WARNING (significant lean trend under boost)
  
  Advisory: At current rate, boost LTFT will exceed +8% in ~1-2 drives.
  Priority investigation: fuel pressure, boost leaks, injector balance.
  ```

### US-131: Implement Injector Duty Cycle Tracking Analysis

**As a** driver reviewing server reports, **I want** IDC trend tracking **so that** I know when to upgrade injectors before running out of fuel headroom.

**Purpose:** Track how close the fuel injectors are to their flow limit so the driver knows when to upgrade.

**Acceptance Criteria:**
- [ ] **Input:**
  - Peak injector duty cycle per drive
  - RPM and boost at peak IDC
  - Ethanol content at time of peak
- [ ] **Process:**
  - Track peak IDC trend over drives
  - Calculate headroom (100% - peak IDC)
  - Project at what boost level IDC will exceed 85% (based on relationship between boost and IDC)
- [ ] **Output format** (exact):
  ```
  INJECTOR DUTY CYCLE REPORT — Last {N} Drives
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Current injectors: {size}cc ({model})
  Current fuel: {fuel_type} (avg {ethanol}% ethanol)
  
  Peak IDC this period: {peak}% at {rpm} RPM, {boost} psi boost
  Average peak IDC: {avg}%
  Trend: {trend}% per drive over last {N} drives
  
  Headroom remaining: {headroom}% (at current peak)
  Projected IDC at {next_boost} psi: ~{projected}% ← {status}
  
  Advisory: {specific recommendation}
  ```
- [ ] Tests validate with Spool's worked example:
  ```
  Current injectors: 550cc (ID550), fuel: E85 (avg 74% ethanol)
  Peak IDC: 82% at 5200 RPM, 14.5 psi boost
  Average peak: 78%, Trend: +1.2%/drive over 5 drives
  Headroom: 18%, Projected at 15 psi: ~87% ← CAUTION
  
  Advisory: Adequate margin at current boost. If planning 15 psi, upgrade to 660cc first.
  ```

### US-132: Implement Thermal Pattern Analysis

**As a** driver reviewing server reports, **I want** ambient-normalized coolant trends **so that** I can detect cooling system degradation before it causes head gasket failure.

**Purpose:** Detect cooling system degradation before it causes a head gasket failure.

**Acceptance Criteria:**
- [ ] **Input:**
  - Coolant temp time-series from each drive
  - Ambient temperature (from IAT at drive start, before engine heat-soaks the intake)
  - Driving pattern (idle time, cruise time, boost time from TPS/RPM data)
- [ ] **Process:**
  - Calculate peak coolant temp per drive
  - Calculate average coolant temp during cruise
  - Normalize for ambient temperature (a 95F day will run hotter than a 70F day)
  - Track normalized trend over drives
- [ ] **Output format — Normal** (exact):
  ```
  THERMAL REPORT — Last {N} Drives
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Ambient-adjusted peak coolant trend:
    Drive {n} ({date}, {ambient}F ambient): Peak {peak}F → Normalized: +{delta}F above ambient
  
  Trend: {trend}F/drive normalized (within normal variation)
  
  Advisory: Cooling system is performing consistently.
    No degradation detected. Peak temps are tracking ambient as expected.
  ```
- [ ] **Output format — Abnormal:**
  ```
  Trend: +1.5F/drive normalized (CONCERNING)
  
  Advisory: Coolant temps are rising faster than ambient can explain.
    Normalized peak is +{delta}F higher than baseline from {weeks} ago.
    Possible causes:
      1. Coolant level low (check overflow tank)
      2. Thermostat starting to stick
      3. Radiator fan relay intermittent
      4. Radiator fins clogged (bugs, debris)
    Action: Check coolant level before next drive. If level is fine, monitor for 2 more drives.
    If trend continues, investigate thermostat and fan operation.
  ```
- [ ] Tests validate ambient normalization and trend detection
- [ ] Tests validate with Spool's normal example:
  ```
  Drive 1 (June 1, 72F ambient): Peak 198F → Normalized +126F
  Drive 5 (June 15, 78F ambient): Peak 203F → Normalized +125F
  Drive 10 (July 1, 85F ambient): Peak 214F → Normalized +129F
  Trend: +0.3F/drive → Normal variation
  ```

### US-133: Implement E85 Content Tracking Analysis

**As a** driver reviewing server reports, **I want** ethanol content tracked per fillup **so that** I know what my local stations are actually selling.

**Purpose:** Track ethanol content consistency from local fuel stations so the driver knows what they're actually getting.

**Acceptance Criteria:**
- [ ] **Input:**
  - Ethanol content readings from flex fuel sensor over each drive
  - Stable readings only (ignore first 5 minutes after fillup as fuel mixes in tank)
- [ ] **Process:**
  - Record average ethanol % per fillup
  - Track by station if driver provides station info (future feature)
  - Alert on significant deviations
- [ ] **Output format** (exact):
  ```
  E85 CONTENT TRACKING — Last {N} Fill-ups
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Fill {n} ({date}): {ethanol}% ethanol
  ...
  
  Average: {avg}% ethanol
  Range: {min}-{max}%
  
  Advisory: {context about E85 pump variability and any action items}
  ```
- [ ] Advisory provides context: "E85 at the pump typically ranges 51-83% depending on season and region."
- [ ] Flags notably low fills (< 65%) with note that ECMLink is compensating
- [ ] Tests validate with Spool's worked example:
  ```
  5 fill-ups: 78%, 74%, 72%, 81%, 65%
  Average: 74%, Range: 65-81%
  Fill 5 flagged as LOW (65%)
  Advisory: Local E85 averaging 74% (normal). Fill 5 notably low.
  ECMLink compensating correctly via flex fuel interpolation. No action needed.
  ```

### US-134: Implement Baseline Comparison Analysis

**As a** driver reviewing server reports, **I want** each drive compared to an established baseline **so that** I can catch anything that's changed.

**Purpose:** Compare today's drive to the established baseline to catch anything that's changed.

**Acceptance Criteria:**
- [ ] **Input:**
  - Current drive data (all parameters)
  - Baseline established from first 5 "healthy" drives after initial tune
- [ ] **Process:**
  - For each parameter at each operating condition (idle, cruise, boost), compare current average with baseline average
  - Flag deviations > 2 standard deviations from baseline
  - Correlate flagged parameters (e.g., high LTFT + high IAT = heat soak, not a real problem)
- [ ] **Output format** (exact):
  ```
  BASELINE COMPARISON — Drive {date} vs Baseline
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Parameters within normal range: {pass}/{total} ✓
  
  Deviations:
    {param} at {condition}: {value} (baseline: {baseline_value}) → +{delta} above baseline
      Context: {explanation — e.g., ambient temp difference}
      Adjusted deviation: {adjusted} — {assessment}
  
  Overall: {summary with any items to watch}
  ```
- [ ] Correlation logic: distinguishes ambient-explained deviations from real deviations
- [ ] Tests validate with Spool's worked example:
  ```
  IAT at cruise: 142F (baseline: 118F) → +24F above baseline
    Context: Ambient 94F vs baseline avg 72F → Adjusted: +2F → Normal
  
  LTFT at idle: +4.8% (baseline: +1.2%) → +3.6% above baseline
    Context: No corresponding ambient change → Real deviation
    Action: Monitor 2 more drives. If LTFT continues >+5% at idle, inspect vacuum lines.
  ```

### US-135: Implement Drive Summary Generation

**As a** driver, **I want** a comprehensive drive summary **so that** I can review each drive's key stats and safety status.

**Purpose:** Generate a summary for driver display (on Pi) and server storage after each drive.

**Acceptance Criteria:**
- [ ] **Input:** Complete drive session data (all parameters, all events)
- [ ] **Output format** (exact):
  ```
  DRIVE SUMMARY — {date}
  ━━━━━━━━━━━━━━━━━━━━━━━━━
  Duration: {duration} | Distance: {distance} mi
  Status: {✓ ALL NORMAL | ⚠ CAUTIONS | ✗ DANGERS}
  
  Engine:
    Coolant: {avg}F avg, {peak}F peak {status}
    IAT: {avg}F avg, {peak}F peak {status}
    RPM: {peak} peak
    Boost: {peak} psi peak
  
  Fueling:
    AFR (WOT): {min}:1 min {status} (target: {target_low}-{target_high})
    Ethanol: {ethanol}%
    IDC: {peak}% peak {status}
    STFT: {avg}% avg {status}
    LTFT: {ltft}% {status}
  
  Safety:
    Knock events: {count} {status}
    Battery: {avg}V {status}
    Alerts: {alert_summary}
  
  {overall_assessment}
  ```
- [ ] Status markers: ✓ (normal), ⚠ (caution occurred), ✗ (danger occurred)
- [ ] Phase 1 version omits Phase 2 fields (AFR, IDC, knock, boost, ethanol)
- [ ] Summary stored in database and displayed in parked mode (B-030 US-128)
- [ ] Tests validate with Spool's worked example:
  ```
  Duration: 1h 15m, Distance: 42 mi, Status: ALL NORMAL
  Coolant: 192F avg, 198F peak ✓
  IAT: 118F avg, 138F peak ✓
  RPM: 6200 peak, Boost: 14.2 psi peak
  AFR (WOT): 11.2:1 min ✓ (target: 11.0-11.5)
  Ethanol: 74%, IDC: 78% peak ✓
  STFT: +2.1% avg ✓, LTFT: +1.5% ✓
  Knock: 0 ✓, Battery: 14.2V ✓
  Assessment: "No issues detected. Happy engine."
  ```

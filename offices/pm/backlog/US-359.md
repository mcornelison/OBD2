# US-359 — Pi-side dual-attribution reproducer harness

## Goal
As a developer hardening the Pi-side drive lifecycle, I want a deterministic unit-test harness that replays the Drive 23/24 timing and signal sequence so that the dual-emission defect is reproduced in CI before the fix lands and is locked-in as a regression net after the fix lands.

## Definition of Done (**at least 1 criterion required** so Ralph knows when complete)
- New test fixture under `tests/pi/obdii/drive/` replays Drive 23/24 timing (starts 3 seconds apart; ~6m 30s overlap window) against the DriveDetector + lifecycle orchestrator integration seam.
- Fixture is deterministic (no wall-clock dependency; uses injected clock or recorded-sample fixture).
- Pre-fix run of the fixture against current `main` HEAD asserts the defect: DriveDetector emits 2 distinct drive_ids on the same physical leg.
- Post-fix run (re-executed by US-361) of the SAME fixture asserts: DriveDetector emits exactly 1 drive_id.
- Fixture wired into `pytest -m "not slow"` default sweep (not gated behind a marker; it is the regression net).
- Atlas reviewer sign-off recorded in PR description (no PM Rule 10 trigger -- test-only).

## Conditional outcomes
- if the unit-test harness alone cannot surface the defect (timing reproduces only under multi-thread emission races), then escalate to in-car drill with thread-id + emission-seam instrumentation per F-107 acceptance criteria Step 1 second bullet.
- if Drive 23/24 raw telemetry from `realtime_data` is no longer accessible on chi-srv-01, then derive synthetic per-second RPM/timestamp sequence from F-107 description constants (RPM samples differ by 1500-2000 between drive_ids at adjacent seconds; combined cadence ~1 sample / 1.55s).

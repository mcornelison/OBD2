## 1. New branching workflow: main = production; dev = integration

**STATUS**: DONE 2026-05-28 -- spec committed `b277f8b` (`docs/superpowers/specs/2026-05-28-dev-main-branching-workflow-design.md`); implementation landed per `docs/superpowers/plans/2026-05-28-dev-main-branching-workflow.md`. PM Rules 8 + 9 rewritten (`8e883f1`); three skills retargeted (`d379025`); `pm_status.py` shows both branch tips (`be99b6c` + `5039d0c` + `ba204c5`); `dev` branch bootstrapped from main (origin/dev exists at the same SHA as origin/main pre-push). Chain concept preserved -- only merge target changes (sprint-branch-tip → dev).

**Current model** (V0.27.X): Sprint branches `sprint/sprint41-bugfixes-V0.27.17` directly off main; deploy from sprint branch; merge to main only after chain-end validation.

**New model** (V0.28.X+):
- `main` = production / fully validated stable (unchanged role)
- `dev` = integration branch (NEW; sits between sprint branches and main)
- Sprint branches branch off `dev`, do their work, merge back to `dev`
- Validation happens off `dev` branch (not off sprint branches directly)
- When dev validates → merge dev to main
- Each subsequent sprint branches fresh from updated `dev`

**Why CIO directs this:** clean separation. main never sees "deployed but not yet validated" state. dev is where integration happens + where validation drills run. Sprint branches stay short-lived.

**PM Rule update owed:** PM Rule 8 (sprint-branch workflow) + PM Rule 9 (validation-gated merge) need rewrites to reflect dev/main two-tier. Atlas review of the workflow change at sprint-spin time (PM Rule 10 design-gate territory if it affects deploy/release infrastructure).

**Skill updates owed:**
- `/sprint-deploy-pm` -- deploy FROM dev (not sprint branch directly) after sprint merges to dev
- `/sprint-validated` -- validates dev; on PASS, merges dev to main
- `/chain-validated` -- may become unnecessary or transform into "/release-validated dev → main"; reconsider

## 2. Validation criteria defined UPFRONT (before sprint work begins)

CIO 2026-05-23: "before the beginning of a dev we need to define the validation criteria too before work has begun. it should be defined as testable actions with outcomes. these will be based on what is in the user story that define that feature release."

**Current state (V0.27.X)**: `sprint.json` has a `validation.bigDefinitionOfDone` field (Sprint 28+ requirement per `sprint_lint.lintSprintValidation`). It's been populated for recent sprints but inconsistently rigorous; some sprints relied on per-story `acceptance` arrays + late-added IRL gates.

**Standard CIO wants (V0.28+)**:
- Each user story has testable validation criteria DEFINED IN THE STORY before Ralph picks it up
- Format: testable action + expected outcome
- Sprint-level `validation.bigDefinitionOfDone` aggregates the user-story criteria (not introduced as new clauses at deploy time)
- Atlas reviews the validation block at sprint-spin time (already happens in Sprint 39/40 cadence; formalize as PM Rule)

**Rationale**: prevents the "feature shipped but never actually tested IRL" pattern that surfaced repeatedly in V0.27.X chain (US-326/328 false-pass, US-348/349 redo false-pass, US-351 schema-vs-ORM mask). Upfront validation criteria pin "done" before code is written, not after.

**Connects to item 1**: validation criteria are tested against `dev` branch before dev → main merge. The criteria definition happens at sprint-spin time on dev.

## 3. Backlog hierarchy -- restore Epic > Feature > User Story 3-tier SSOT

**Current state**: `offices/pm/backlog.json` has the right structure (epics with features underneath; story counter for IDs) -- per Sprint 8 PMO migration. But it's STALE:
- `lastUpdated: 2026-05-11` (per backlog.json header)
- Missing ~30+ B-XXX items added since (B-076 through B-109)
- Missing all the V0.28+ candidates filed in recent grooming sessions

Individual `B-XXX.md` files in `offices/pm/backlog/` are flat (not file-system-organized by epic).

**CIO direction 2026-05-23**: "I would like a hierarchy of three tiers. epic, feature and user stories. an example of epic might be user interface, and an example of a feature would be the server cli, and pi display, and a user story would be the details of I want to... x to see y..."

**Standard user-story formats** (research per CIO):
- **Connextra / classic**: `As a <role>, I want <goal> so that <benefit>` -- most common; concise
- **Gherkin / BDD**: `Given <precondition>, when <action>, then <outcome>` -- testable directly; pairs well with item 2 (validation-criteria-upfront)
- **Job stories**: `When <situation>, I want to <motivation>, so I can <expected outcome>` -- focuses on context

CIO's example phrasing ("I want to x to see y, if x happens the y,z then happened") suggests hybrid -- some Connextra + some Gherkin. Recommend supporting both formats; let story drafter pick the one that captures the testable shape better.

**Work owed at V0.28.0 grooming**:
- Re-sync `backlog.json` with all current B-XXX items (catch up the ~30+ missing)
- Verify epic → feature relationships are correct (some current B-XXX may need re-parenting)
- File system reorg decision: keep flat `B-XXX.md` files? OR move to `pm/backlog/E-XX/B-YYY.md`? OR add view file `pm/backlog-hierarchy.md` as derivative? **PM lean: keep flat files + use backlog.json as SSOT for hierarchy view; less disruption.**
- Sprint 41 `validation.bigDefinitionOfDone` already has examples of testable clauses; use them as templates for V0.28.0 sprint contracts
- Update `_template.md` in `pm/backlog/` to require user-story format (Connextra or Gherkin)

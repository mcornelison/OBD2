## Description

Validate and update the existing tiered PID polling configuration against Spool's tuning-driven priority tiers. The system already has a tiered polling concept (from obd2-research.md), but Spool has refined the tiers based on tuning priorities. Also includes Phase 2 data architecture planning for the 30x increase in data volume when ECMLink is installed.

**Source of truth**: Polling priorities from Spool (Tuning SME). Data volume estimates from Spool.

## User Stories

### US-136: Validate Phase 1 Tiered Polling Configuration

**As a** developer, **I want** the polling tiers to match Spool's tuning-driven priorities **so that** safety-critical parameters are always fresh.

**Acceptance Criteria:**
- [ ] Verify/update polling config to match Spool's 4-tier structure:

  **Tier 1 — Every cycle (1 Hz):** Core 5 PIDs
  | PID | Name | Why It's Tier 1 |
  |-----|------|-----------------|
  | 0x05 | Coolant Temperature | Engine protection. #1 priority. Head gasket fails above 220F. |
  | 0x0C | Engine RPM | Context for every other reading. Defines engine state. |
  | 0x04 | Calculated Engine Load | Workload context. High load + high temp = danger. |
  | 0x0E | Timing Advance | Only knock indicator without ECMLink. Dropping timing = possible detonation. |
  | 0x06 | Short-Term Fuel Trim B1 | Best real-time lean/rich indicator. Large positive = engine adding fuel = something lean. |

  **Tier 2 — Every 3 cycles (~0.3 Hz):**
  | PID | Name | Why |
  |-----|------|-----|
  | 0x11 | Throttle Position | Load context. Distinguishes idle vs cruise vs WOT. |
  | 0x0D | Vehicle Speed | Context. Stationary vs moving. Gear estimation. |

  **Tier 3 — Every 10 cycles (~0.1 Hz):**
  | PID | Name | Why |
  |-----|------|-----|
  | 0x07 | Long-Term Fuel Trim B1 | Trend indicator. LTFT drifting positive over weeks = developing problem. |
  | 0x0F | Intake Air Temperature | Charge density context. High IAT = less power, more knock risk. |
  | 0x14 | O2 Sensor B1S1 | Narrowband O2. Should oscillate 0.1-0.9V. Stuck = dead sensor. |

  **Tier 4 — Every 30 cycles (~0.03 Hz):**
  | PID | Name | Why |
  |-----|------|-----|
  | 0x42 | Control Module Voltage | Battery/charging health. Drop below 13.5V = alternator issue. |
  | 0x0B | Manifold Pressure (MDP) | **CAUTION: This is MDP, not true MAP on this car.** Use for relative reference only, NOT boost measurement. |

- [ ] Config file reflects all 4 tiers with correct cycle counts
- [ ] Tier rationale documented in config comments or accompanying doc
- [ ] MDP caveat captured: PID 0x0B is MDP (EGR only) on 2G Eclipse, NOT true manifold pressure — cannot be used for boost measurement
- [ ] Tests validate polling frequency matches tier assignments

### US-137: Define ECMLink Parameter Ingestion Schema (Phase 2)

**As an** architect, **I want** the database schema and ingestion pipeline designed for ECMLink data **so that** Phase 2 integration is straightforward when hardware is ready.

**Blocked**: Cannot test with real data until ECMLink installed. Schema design and stub implementation can proceed.

**Acceptance Criteria:**
- [ ] ECMLink parameter schema defined for 15 priority parameters:
  | Parameter | Unit | Sample Rate | Data Type |
  |-----------|------|------------|-----------|
  | Wideband AFR | ratio (e.g., 11.5:1) | 20 Hz | float |
  | Knock Count | count | 20 Hz | integer |
  | Knock Sum | cumulative | 20 Hz | integer |
  | Boost (MAP) | psi | 20 Hz | float |
  | Injector Duty Cycle | % | 10 Hz | float |
  | Timing Advance | degrees BTDC | 20 Hz | float |
  | RPM | RPM | 20 Hz | integer |
  | TPS | % | 20 Hz | float |
  | Coolant Temp | F | 5 Hz | float |
  | IAT | F | 5 Hz | float |
  | Ethanol Content | % | 1 Hz | float |
  | Target AFR | ratio | 10 Hz | float |
  | STFT | % | 10 Hz | float |
  | LTFT | % | 1 Hz | float |
  | Barometric Pressure | kPa | 0.5 Hz | float |
- [ ] Database table(s) designed to handle mixed sample rates
- [ ] Schema documented in `specs/architecture.md`
- [ ] Ingestion interface defined (how does ECMLink serial data enter the system?)

**From Spool**: "ECMLink provides a serial data stream at 10-50 Hz with 50+ parameters. This is a completely different data volume. The PM and architect need to plan for this NOW even if it ships later."

### US-138: Design Data Volume Architecture for Phase 2

**As an** architect, **I want** storage and retention strategy designed for Phase 2 data volumes **so that** the system handles ECMLink's 30x data increase.

**Blocked**: Architecture design can proceed; implementation blocked on ECMLink.

**Acceptance Criteria:**
- [ ] Data volume estimates documented and accounted for:
  - **Phase 1 (OBD-II)**: ~5 readings/sec × 3600 sec/hr = ~18,000 rows per hour of driving
  - **Phase 2 (ECMLink)**: ~15 params × 10 Hz avg = ~150 readings/sec = ~540,000 rows per hour
  - **Typical summer weekend drive**: 1-3 hours
  - **Seasonal estimate**: ~20 drives × 2 hrs avg = ~40 hours = ~21.6 million rows/season on ECMLink
- [ ] Pi SQLite storage strategy: can Pi store a full season? (estimate disk usage per million rows)
- [ ] Chi-Srv-01 MariaDB strategy: retention policy, partitioning, indexing for 21M+ rows/season
- [ ] Sync bandwidth estimate: how long does it take to sync a 2-hour ECMLink drive over WiFi?
- [ ] Architecture decisions documented in `specs/architecture.md`
- [ ] Retention policy updated: 90 days on Pi (existing), forever on server (existing) — validate these still work at Phase 2 volumes

## Notes

- Phase 1 polling (US-136) is buildable and testable NOW with the simulator.
- Phase 2 stories (US-137, US-138) are architecture/design work — schemas and docs, not runtime code. They can proceed without hardware.
- The MDP caveat (PID 0x0B is NOT true MAP on the 2G Eclipse) is critical — this has tripped up DSM builders before.

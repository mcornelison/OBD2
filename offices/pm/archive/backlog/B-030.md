## Description

Implement the tuning-driven display content and layout for the OSOYOO 3.5" HDMI touchscreen (480x320 pixels). This defines WHAT the driver sees — the primary driving screen, 5 detail pages, touch interactions, and parked mode. B-007 handles the hardware layer (GPIO, I2C, UPS); this item handles the tuning-informed content.

**Design principle (from Spool):** The driver is DRIVING. Information must be glanceable in under 1 second. No text paragraphs. No menus while moving.

**Source of truth**: Display specs from Spool (Tuning SME). Layout decisions are UI designer's domain; Spool specifies the elements and priorities.

## User Stories

### US-121: Implement Primary Driving Screen with Status Indicator

**As a** driver, **I want** a glanceable primary screen with an overall health indicator **so that** I can see "is everything OK?" in under 1 second.

**Acceptance Criteria:**
- [ ] Primary driving screen is the default view when engine is running
- [ ] Display elements with priority:
  | Element | Size Priority | Update Rate | Format |
  |---------|--------------|-------------|--------|
  | **System Status Indicator** | Large, always visible | Real-time | Green circle / Yellow triangle / Red X |
  | **Coolant Temperature** | Large | 1 Hz | "185F" — changes color at thresholds |
  | **RPM** | Medium | 1 Hz | "2500" or bar graph |
  | **Boost** (Phase 2 only) | Medium | Real-time | "12.3 psi" — only shown when MAP sensor installed |
  | **AFR** (Phase 2 only) | Medium | Real-time | "11.5" — only shown when wideband installed |
- [ ] **Status indicator is the most important element:**
  - **Green**: All parameters within normal range. Drive happy.
  - **Yellow**: One or more parameters in caution range. Tap for details.
  - **Red**: One or more parameters in danger range. Audible alert. Tap for details.
- [ ] Phase 2 elements (Boost, AFR) hidden when hardware not present — screen adapts
- [ ] All values change color at threshold boundaries (from B-028/B-029)
- [ ] Screen renders correctly on 480x320 resolution
- [ ] Tests validate status indicator logic (worst-parameter-wins for overall status)

### US-122: Implement Thermal Detail Page

**As a** driver, **I want** a thermal detail page **so that** I can see temperature trends and time-at-temperature.

**Acceptance Criteria:**
- [ ] Thermal page shows:
  - Coolant temp with trend arrow (stable, rising, falling)
  - Intake air temp
  - Time at temperature: how long has coolant been above 200F this drive?
- [ ] Trend arrow calculated from last 60 seconds of readings:
  - Rising: average slope > +0.5F/min
  - Falling: average slope < -0.5F/min
  - Stable: between -0.5 and +0.5F/min
- [ ] Time-at-temperature counter resets per drive
- [ ] Accessible by swiping from primary screen
- [ ] Tests validate trend calculation and time-at-temp accumulation

---

### US-123: Implement Fuel Detail Page (Phase 2)

**As a** driver, **I want** a fuel system detail page **so that** I can monitor AFR, fuel trims, and injector load.

**Acceptance Criteria:**
- [ ] Fuel page shows (Phase 2 — ECMLink required):
  - AFR actual vs target (side by side or overlay)
  - STFT / LTFT values
  - Injector duty cycle (with color at threshold boundaries)
  - Ethanol content % (from flex fuel sensor)
- [ ] Hidden/unavailable when ECMLink not connected
- [ ] All values color-coded at their respective thresholds
- [ ] Tests validate layout with mock ECMLink data

---

### US-124: Implement Knock Detail Page (Phase 2)

**As a** driver, **I want** a knock monitoring page **so that** I can see detonation events and context.

**Acceptance Criteria:**
- [ ] Knock page shows (Phase 2 — ECMLink required):
  - Knock count this drive
  - Knock sum this drive
  - Last knock event details: RPM, load, timing at moment of knock
- [ ] Hidden/unavailable when ECMLink not connected
- [ ] Last knock event updates in real-time when a new knock occurs
- [ ] Tests validate knock event display with mock data

---

### US-125: Implement Boost Detail Page (Phase 2)

**As a** driver, **I want** a boost monitoring page **so that** I can track boost pressure and detect wastegate issues.

**Acceptance Criteria:**
- [ ] Boost page shows (Phase 2 — ECMLink required):
  - Current boost pressure
  - Target boost pressure
  - Peak boost this drive
- [ ] Peak boost resets per drive
- [ ] Color coding at boost threshold boundaries (B-029)
- [ ] Hidden/unavailable when ECMLink not connected
- [ ] Tests validate peak tracking and display

---

### US-126: Implement System Detail Page

**As a** driver, **I want** a system status page **so that** I can see Pi health, connectivity, and drive stats.

**Acceptance Criteria:**
- [ ] System page shows:
  - Pi battery level (from UPS when available)
  - WiFi status (connected/disconnected, SSID if connected)
  - Last sync time (to Chi-Srv-01)
  - OBD connection status (connected/disconnected/simulated)
  - Drive duration (current drive elapsed time)
- [ ] Always available (not Phase 2 dependent)
- [ ] Tests validate display with various connection states

---

### US-127: Implement Touch Interactions and Safety Constraints

**As a** driver, **I want** safe, simple touch interactions **so that** I can navigate without distraction.

**Acceptance Criteria:**
- [ ] Touch gestures implemented:
  | Gesture | Action |
  |---------|--------|
  | Tap status indicator | Show details of current alert |
  | Swipe left/right | Cycle between detail pages |
  | Tap and hold (3 sec) on alert | Acknowledge / dismiss non-critical alert |
- [ ] **SAFETY CONSTRAINT: No interactions while vehicle speed > 0 that change system behavior.** Read-only while driving. Touch only navigates between display pages.
- [ ] Page order: Primary → Thermal → Fuel → Knock → Boost → System (swipe cycles)
- [ ] Phase 2 pages skipped when ECMLink not connected
- [ ] Tests validate gesture recognition and safety constraint enforcement

---

### US-128: Implement Parked Mode Display

**As a** driver, **I want** a parked mode screen **so that** I can see sync status and drive summary when the engine is off.

**Acceptance Criteria:**
- [ ] Parked mode activates when engine is off and Pi is running on battery
- [ ] Parked mode shows:
  - Sync status: "Syncing to server... 45%" or "Sync complete"
  - Last drive summary: duration, distance, any alerts
  - Server advisory messages (if any downloaded from last sync)
  - Battery remaining: "Pi battery: 78% (~1.5 hrs)"
- [ ] Transitions automatically when engine off detected (RPM = 0 sustained)
- [ ] Returns to primary driving screen when engine restarts
- [ ] Tests validate mode transitions and display content

## Notes

- B-007 (Touch Screen Display Support) provides the hardware layer (I2C, GPIO, display init). This item provides the content layer.
- Phase 2 pages (Fuel, Knock, Boost) should be designed now but hidden until ECMLink data is available.
- The primary screen status indicator aggregates ALL parameter alerts — worst status wins.
- Display adapts based on available hardware: Phase 1 shows fewer elements, Phase 2 adds boost/AFR/knock.
- Spool is available for display content questions — `offices/tuner/inbox/`.

### Sweep 2a Lessons Learned (apply to Sweep 2b+)
1. **Audit before assumption**: the audit saved the sweep. The original plan assumed the easy path; the audit found the blocker. Always do Task 2-style classification before touching code.
2. **Blast radius larger than audit predicted**: audit said ~17 test files reference legacy keys. Actual failure count after rewire was 127 tests across 14 files because creating the AlertManager raises through every test that calls `orchestrator.start()`. Audit the **exercise path**, not just the **reference path**, when predicting regression scope.
3. **Rewire-before-delete is the only safe split**: you cannot do a data-model cleanup before the consumer of that data is rewired. Sweep 2b is finally a pure delete because 2a did the logic change first.
4. **File descriptor hygiene in tests**: the code reviewer caught a bare `open()` in a Task 3 test. `with configPath.open() as f:` is the correct pattern. Grep new test files for `open(` before committing.
5. **Never hardcode absolute project paths in tests**: `pathlib.Path(__file__).resolve().parent.parent / 'src' / 'obd_config.json'` is the canonical resolve pattern. `Z:/o/OBD2v2/...` will break on the Pi (`/home/mcornelison/Projects/EclipseTuner`) and any other dev machine.
6. **Modification History discipline**: `specs/standards.md` requires an entry per modification. The reviewer caught that I missed it on Task 3's manager.py. Add a row whenever you change a file's public API.
7. **Don't use compound shell commands** (`cd && grep`). In this project, chains trigger per-chunk permission prompts even when each command is individually pre-approved. Use `git -C`, absolute paths, and parallel Bash calls. Saved to `feedback_no_compound_bash.md`.

## Sweep 1 Lessons Learned (apply to Sweep 2+)
1. Path convention: top-level packages (display/alert/analysis/profile/power/calibration/ai/backup/hardware/common) import WITHOUT `src.` prefix. Reason: tests/conftest.py puts src/ itself on sys.path. Using `from src.display import` breaks subprocess tests. Use bare `from display import`.
2. Option A pattern: when a plan nominates a submodule as canonical (e.g., config/loader.py) but symbols actually live across the package (loader + helpers + simulator), the correct canonical is the PACKAGE __init__.py not the submodule. Check before rewriting.
3. Lazy imports inside functions need both: (a) the actual import rewritten AND (b) every @patch() target in tests that mocks that lazy call. Grep tests for old-facade-name patch strings.
4. @patch('obd.X.createFoo') where X is a facade — after rewiring to canonical package, patch target must be the new source module. Example: @patch('alert.createAlertManagerFromConfig') not 'obd.alert_manager.createAlertManagerFromConfig'.
5. sys.modules dict patches (patch.dict('sys.modules', {'obd.X': MagicMock(...)})) also need the key renamed to the new module path.
6. Parallel-session branch gotcha: if PM or another agent is working in parallel on the same repo and does `git checkout main` while Ralph is on a sprint branch, Ralph's working tree flips too. Recover via `git stash push -u`, re-checkout the sprint branch, `git stash pop`. Ralph's edits survive. Not data loss, just confusing.
7. Ruff gate: always run `ruff check src/ tests/` at Task 8. Ruff will flag I001 import-ordering issues introduced by rewrite — auto-fix via `ruff check --fix --select I001 <file>` on the specific files, NOT the whole tree (avoid fixing pre-existing unrelated warnings).
8. mypy may not be installed in the local env — task 8 mypy step is best-effort, skip if not available.

## Codebase Patterns
- ConfigValidator uses dot-notation for nested key access (e.g., 'database.server')
- Default values are defined in module-level DEFAULTS dict
- Custom exceptions should include typed field lists for clear debugging
- All modules in src/common/ follow camelCase for functions, PascalCase for classes
- Test files use AAA pattern (Arrange, Act, Assert) with descriptive docstrings
- Use relative imports within subpackages (e.g., `from .types import ...`)
- Use absolute imports across subpackages (e.g., `from src.obd.data.types import ...`)
- Types modules should have no dependencies on other project modules
- Consider using `TYPE_CHECKING` for type hints to avoid runtime circular imports
- Lazy imports (inside functions): patch at source module `@patch('obd.orchestrator.X')`, not caller module
- Gotcha: Windows subprocess pipes have ~4096 byte buffer — redirect stdout/stderr to files for verbose subprocess tests
- Application logs go to stdout (StreamHandler(sys.stdout)), not stderr — check stdout for log analysis in tests
- Tiered thresholds config lives in obd_config.json under `tieredThresholds` (separate from single-threshold `alertThresholds` in profiles)
- Tiered threshold boundary semantics: value > boundary triggers higher level; value == boundary stays in lower level
- Bidirectional thresholds (STFT): use abs(value) >= threshold, then check sign for lean/rich message selection
- Baseline-relative thresholds: use a stateful tracker class, update baselines only during normal readings, not during retard events
- RPM/load banding: int(value // bandSize) * bandSize for floor-based grouping into operating point buckets
- Asymmetric bidirectional thresholds (Battery Voltage): check danger-high first (>), then danger-low (<=), then normal range, else caution with low/high message selection based on which side of normal
- Sensor failure detection (IAT): track consecutive readings at a sentinel value (-40F), only flag after configurable N consecutive hits; reset counter on any non-sentinel reading
- When tiered_thresholds.py exceeds ~300 lines, create separate module (e.g., iat_thresholds.py, timing_thresholds.py) importing shared types from tiered_thresholds
- Polling tier config uses tier1/tier2/tier3/tier4 keys in obd_config.json pollingTiers section; load with src/obd/data/polling_tiers.py loadPollingTiers()
- Cycle scheduling: cycleNumber % cycleInterval == 0 determines poll eligibility per tier

## Orchestrator Deep Dive (2026-01-29)

### Structure: 2,500 lines, 1 god class, 70+ methods, 40+ instance vars, 12 component dependencies
### Supporting: ShutdownState enum, HealthCheckStats dataclass, 4 exception classes, 1 factory function

### 12 Responsibility Areas
1. Lifecycle (init/start/stop) - 500 lines
2. Signal handling (SIGINT/SIGTERM, double-Ctrl+C) - 100 lines
3. Main event loop (100ms tick, health checks) - 100 lines
4. Event routing (callbacks for readings, drives, alerts, analysis, profiles) - 400 lines
5. Connection recovery (exponential backoff, pause/resume data logger) - 350 lines
6. Health monitoring (stats, data rate, uptime) - 200 lines
7. Status reporting (component status, backup status) - 130 lines
8. Display updates (distributed across callbacks)
9. VIN decoding (first-connection query, NHTSA API) - 120 lines
10. Backup management (init, catchup, schedule, upload, cleanup) - 300 lines
11. Hardware integration (Pi-only, UPS, GPIO) - distributed
12. Config extraction (timeouts, intervals, parameters) - 90 lines

### Component Init Order (dependency chain)
Database -> ProfileManager -> Connection -> VinDecoder -> DisplayManager -> HardwareManager -> StatisticsEngine -> DriveDetector -> AlertManager -> DataLogger -> ProfileSwitcher -> BackupManager

### Shutdown Order (reverse)
BackupManager -> DataLogger -> AlertManager -> DriveDetector -> StatisticsEngine -> HardwareManager -> DisplayManager -> VinDecoder -> Connection -> ProfileSwitcher -> ProfileManager -> Database

### Data Flow
Reading: DataLogger -> Orchestrator._handleReading -> DisplayManager + DriveDetector + AlertManager
Drive: DriveDetector -> Orchestrator._handleDriveStart/End -> DisplayManager + external callback
Alert: AlertManager -> Orchestrator._handleAlert -> DisplayManager + HardwareManager + external
Analysis: StatisticsEngine -> Orchestrator._handleAnalysisComplete -> DisplayManager + external
Profile: ProfileSwitcher -> Orchestrator._handleProfileChange -> AlertManager + DataLogger

### Proposed 7-Module Split
- types.py (~100 lines): ShutdownState, HealthCheckStats, exceptions, constants
- core.py (~750 lines): ApplicationOrchestrator class, __init__, properties, getStatus, runLoop
- lifecycle.py (~400 lines): All _initialize*() and _shutdown*() methods
- connection_recovery.py (~350 lines): Reconnection logic, pause/resume
- event_router.py (~400 lines): Callback registration and all _handle*() methods
- backup_coordinator.py (~300 lines): Backup init, scheduling, upload
- health_monitor.py (~200 lines): Health checks, stats, data rate
- signal_handler.py (~100 lines): SIGINT/SIGTERM handling

### Good Patterns Found
- Graceful fallbacks (HARDWARE_AVAILABLE, BACKUP_AVAILABLE flags)
- Dependency-ordered init, reverse-order shutdown
- Timeout-based shutdown with force-exit
- Exponential backoff for reconnection
- Double-Ctrl+C pattern (graceful then force)
- Non-critical errors logged as warnings, don't crash

### Concerning Patterns Found
- God class (2,500 lines, 12 deps, 40+ vars, 70+ methods)
- Component types are Optional[Any] - loses type safety
- 30+ hasattr() calls for duck typing instead of protocols/ABCs
- Threading without full coordination (daemon flag relied on)
- Callback chains hard to trace (reading triggers 4 handlers)
- Some magic numbers inline (0.1 sleep interval)

### Learnings for future iterations:
- **Pattern discovered**: When a state-machine has BOTH a "current state" (mutates both directions via hysteresis) AND a "stages emitted" tracker (one-way per event), keep them as TWO separate attributes. Conflating them forces a choice between hysteresis-de-escalation (which the spec invariant required) and stage-latching (which the spec story required); two attributes lets both behaviors coexist orthogonally. The escalation gates check the latch; the de-escalation path only mutates current state. AC-restore (the "drain event boundary" event) resets BOTH.
- **Pattern discovered**: For TDD on a state-machine bug whose failure mode is "fires too many times", structure tests as two pairs: (a) DISCRIMINATOR tests that count stage-callback invocations + drain-event row counts and FAIL pre-fix; (b) GUARDRAIL tests for the invariants the fix must NOT break (TRIGGER atomicity, AC-restore reset enabling re-fire, single-tick fall-through firing each stage exactly once). The discriminator pair pins the bug class; the guardrail pair pins what the fix is NOT allowed to break. Verify red phase BEFORE implementing -- the "tests pass against pre-fix code" anti-pattern would have shipped a placebo fix.
- **Pattern discovered**: The post-fix IMMINENT gate `_highWaterStage in (NORMAL, WARNING)` is more permissive than the pre-fix `_state == WARNING` gate -- it lets IMMINENT fire from `_state == NORMAL` IF the high-water mark is at WARNING (i.e. after a hysteresis de-escalation). This is the RIGHT behavior under monotonic stage progression: skipping a stage on the way up (because the latch blocks WARNING from re-firing) is fine; what we forbid is re-firing a stage already entered. The test `test_imminent_fires_once_after_warning_de_escalation_then_drop` pins this forward-escalation contract explicitly so a future regression that tightens the gate back to `_state == WARNING` is caught.
- **Gotcha**: The existing US-234 hysteresis test `test_warning_then_recovery_above_band_returns_to_normal` (in `test_orchestrator_vcell_hysteresis.py`) asserts `state == PowerState.NORMAL` after the recovery tick. Under the post-US-288 design `state` is still NORMAL but `_highWaterStage` stays at WARNING -- the test does NOT inspect `_highWaterStage` so it continues to pass. If a future story adds an invariant on `_highWaterStage` after de-escalation, that test may need updating. For now the asymmetry (state can move both directions; high-water is one-way per drain) is the intended design.
- **Gotcha**: TRIGGER's atomicity is now defended by THREE independent gates: (1) the existing `_state == TRIGGER` terminal early-return at the top of `_tickBody`; (2) the existing `_shutdownFired` flag in `_enterTrigger`; (3) the new `_highWaterStage in (NORMAL, WARNING, IMMINENT)` gate. Any one failing alone would still gate. Defense in depth is intentional per Spool's "TRIGGER atomicity unchanged" invariant -- the new high-water gate adds a safety layer without removing or weakening the existing two.
- **Gotcha**: `--check-feedback` sprint_lint flags US-284/285/286/287 for missing commits. This is the standing pattern from prior sessions -- per CIO directive Ralph never runs git commands; PM owns staging. Same as Sessions 158-161 close states; documented as "PM commit-pending" not US-288 territory. US-288's own commits will land at sprint close alongside the rest.

### Learnings for future iterations:
- **Pattern discovered**: PHANTOM-PATH DRIFT documented TWICE in one story (BL-010 pattern). Story scope listed `src/pi/obdii/data/logger.py` for an idempotent `start()` -- but `start()` lives on `RealtimeDataLogger` in `data/realtime.py`, not on `ObdDataLogger` in `data/logger.py` (the latter is the per-query workhorse with no thread lifecycle). - **Pattern discovered**: Defensive type-check in `_readDataLoggerLastRowSecondsAgo` (`isinstance(value, (int, float)) and not isinstance(value, bool)`) caught a regression in `tests/test_orchestrator_loop_health.py::test_performHealthCheck_updatesStatsFromComponents`. The test uses `MagicMock` for `_dataLogger`, and `getattr(magicmock, 'lastRowWrittenSecondsAgo', None)` returns a Mock object (NOT None -- MagicMock auto-creates attributes), which crashed the `f"{lastRow:.1f}"` formatter with `TypeError: unsupported format string passed to MagicMock.__format__`. The type-check converts auto-mocks to 'never_written' which is the correct semantic. **Standing rule**: any new attribute read in production code that existing tests mock with MagicMock needs an isinstance type-check at the read site, or the auto-attribute will return a Mock object that bombs downstream operations. Caught only by running the existing test, not by writing the new test.
- **Pattern discovered**: When implementing a "post-mortem signal" property (`lastRowWrittenSecondsAgo`, `last_attempt_seconds_ago`, etc.), the rendered output in observability code (logs / health checks) MUST use an explicit string sentinel (`never_written`, `n/a`, `none_yet`) for the no-data-yet case, NEVER NULL/None/NaN/-1. This is the no-magic-numbers invariant in the story and a load-bearing operability rule -- explicit sentinels are greppable + unambiguous + survive logger formatter quirks. The dataclass field can hold None, but the render layer translates it.
- **Pattern discovered**: When a story scope says "make X idempotent," audit X first -- it may already be idempotent for the actual scenario at hand. `RealtimeDataLogger.start()` was already MOSTLY idempotent (returns False on RUNNING/STARTING; raises on conn-down). The Spool BUG-2 scenario (lifecycle init start fails, state stays STOPPED, `_handleConnectionRestored` later fires with conn up) is exactly the case the existing state machine handles correctly -- because the conn-check happens BEFORE state mutation, the failed start leaves state at STOPPED (not in some "FAILED" state), so a later start with conn up succeeds. Did NOT add a 'failed-state' enum value (would have been over-engineering; the test `test_start_calledAfterFailedStart_succeedsWhenConnectionReturns` proves the existing state machine works). The bug was 100% in the WIRING, not in start() itself.
- **Pattern discovered**: When wiring a handler to call a downstream component method, do the wiring AFTER the existing display+hardware updates and BEFORE the external user callback. Two reasons: (a) display/hardware updates are local + fast; the new wiring may take longer (start() spawns a thread) -- run cheap state propagation first; (b) the new wiring should win the race against the external callback so the system reaches steady state before user code observes the event. Mirror this in any future "wire X into existing event handler" story.
- **Gotcha**: When running pytest with `-q` mode, output is buffered until the suite finishes. A background pytest run looks "stuck" with 0 bytes of output until the very end, when the entire pass/fail summary lands at once. Use `-v` for streaming output or just trust that a 0-byte file means "still running" not "crashed."
- **Gotcha**: `ObdDataLogger` (in `data/logger.py`) and `RealtimeDataLogger` (in `data/realtime.py`) are easy to confuse -- the former is the per-query workhorse (queryParameter / logReading) that doesn't have a thread lifecycle; the latter is the higher-level wrapper with start()/stop() that owns the polling thread + composes an `ObdDataLogger` as `self._dataLogger`. Story scopes have repeatedly listed `logger.py` when they meant `realtime.py`. The orchestrator's `_dataLogger` attribute holds a `RealtimeDataLogger`.

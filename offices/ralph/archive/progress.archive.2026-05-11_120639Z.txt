## 2026-05-08 - V0.27.1 hotfix (Rex / Session 180)
Task: ObdConnection.connect() thread safety + heartbeat in-flight skip (Sprint 27 engine-on test #2 RCA -> hotfix -> deployed -> verified)

### What was implemented:
- **RCA**: Sprint 27 engine-on test #2 (2026-05-08 17:08 CDT) produced ZERO `realtime_data` rows. Spool diagnosed "stacking-connect bug"; my dev-side RCA from the production journal + thread census + source code reading confirmed the root cause: `obd_connection.connect()` had ZERO thread safety (no `Lock`, no `RLock`). TWO concurrent invokers existed post-Sprint-27: (1) the Sprint 25 `_runInitialConnectWithTimeout` leaked daemon (post-30s-timeout, the worker thread keeps running per the comment at `lifecycle.py:522`); (2) the Sprint 27 US-301 heartbeat's per-tick `_attemptConnectWithCap` daemons (one per tick that timed out, daemon=True so they linger). Both call `self._connection.connect()` directly. With the inner 6-attempt-with-backoff loop, multiple call stacks interleave their attempt counters, all opening `/dev/rfcomm0`, pyserial rejects each open with the literal "device disconnected or **multiple access on port?**" hint. Result: zero successful connects, zero `realtime_data` rows.

### Learnings for future iterations:
- **Pattern discovered**: thread-safety lock at the locked-method's class, not at the heartbeat call site. Spool's first instinct was to put a `threading.Lock()` in `reconnect_loop.py` to serialize the heartbeat's calls. That would only protect heartbeat-vs-heartbeat collision. Putting the lock in `ObdConnection.connect()` itself protects ALL callers (heartbeat, leaked initial-connect daemon, `BtResilienceMixin.handleCaptureError`, `connection_recovery._resumeDataLogging`, every future caller) for free. Standing rule: when the SHARED RESOURCE is owned by class X, the lock belongs in X, not in any one of its callers.
- **Pattern discovered**: lock-state-as-observable-seam. Exposing `isConnectInFlight()` as a public method (returning `self._lock.locked()`) lets external schedulers (heartbeat, watchdogs, retry loops) make scheduling decisions WITHOUT having to acquire the lock themselves. The probe is racy by design (lock state can change between probe-and-call) but for a heartbeat that's about to LOG a missed tick anyway, racy is fine -- worst case the heartbeat blocks briefly on the lock. Cleanest decoupling between resource ownership and scheduler coordination.
- **Pattern discovered**: dual ABI/back-compat for added optional kwargs. `runReconnectHeartbeat` adds `inFlightProbeFn` as a keyword-only param defaulting to None. Every existing test + callsite that doesn't pass it keeps working unchanged. The legacy behavior is the `None` branch (probe-skip never fires). New callsites opt-in by passing the closure. Zero breaking changes.
- **Pattern discovered**: synthetic stronger than empirical for runtime-validation discriminators. Production journal showed 3-4 concurrent connect stacks (the pile-up is bounded by the ratio of inner-cycle-time to heartbeat-tick-interval). The synthetic test using `threading.Barrier(N).wait()` forces N=8 threads to enter simultaneously -- a much stronger signal than letting the race manifest stochastically. Pre-fix RED clearly showed `maxConcurrent=8 violations=7`. Mirror the Barrier idiom for any future thread-safety regression.
- **Gotcha**: `_attemptConnectWithCap`'s daemon thread leaks on timeout by design (Python has no thread cancel; the `daemon=True` ensures process exit reaps them). Pre-V0.27.1 this was fine in isolation. Post-V0.27.1 with the in-flight probe, the heartbeat NEVER spawns when a worker is already running, so the leak window collapses to "at most one heartbeat-spawned worker at a time" + "at most one Sprint-25-leaked initial-connect daemon" = bounded thread count. The fix preserves the daemon-leak architecture but eliminates the unbounded growth pathology.
- **Gotcha**: deploy-pi.sh's release-record schema enforces `theme` <= 50 chars and `description` <= 400 chars. My first two attempts at `RELEASE_VERSION` exceeded both limits (62/779 chars then 31/779). Lesson: when writing a release record for V0.X.Y hotfix descriptions, draft tersely upfront -- the verbose version belongs in this progress.txt entry + the JSON description, not in the deploy schema. Standing recommendation: deploy-pi.sh could pre-validate `RELEASE_VERSION` length BEFORE the rsync + venv steps, since it currently fails at the very last step after burning ~30s of deploy work.
- **Gotcha**: function-name-in-log shifts when refactoring a public method into a thin wrapper. Pre-V0.27.1: journal showed `pi.obdii.obd_connection | connect | Connection attempt 1/6 failed`. Post-V0.27.1: shows `pi.obdii.obd_connection | _performConnect | Connection attempt 1/6 failed`. The `_performConnect` rename is itself a deploy-confirmation marker (cheaper than checking `.deploy-version`) and a useful grep token for V0.27.1+ logs. Worth keeping in mind for future thin-wrapper refactors -- the function name is part of the log signature and changes the journal-walk surface.
- **Gotcha**: when engine is off and OBDLink is unpowered, the BT layer still establishes `/dev/rfcomm0` (paired/bonded since Session 23) but each `obd.OBD()` ctor wedges ~30s before pyserial gives up reading. The leaked Sprint 25 daemon's full 6-attempt cycle therefore takes ~140s on engine-off vs ~5s expected when adapter is powered. NOT a V0.27.1 concern -- it's the latent Sprint 25 architectural debt (TD-047 already filed: "orchestrator init may need subprocess isolation for python-obd library blocks"). V0.27.1 just makes this WEDGE BEHAVIOR observable + non-destructive (heartbeat reports `already_in_flight` continuously instead of stacking competing connects).

- **Pattern: phantom-path wiring drift ONE LAYER UP from the hypothesized root cause.** Original Sprint 28 grooming hypothesized regression in drive_summary.py / detector.py; the actual gap was in realtime.py's missing public method + lifecycle.py's hasattr() probe failing silently. - **Pattern: hasattr() as a silent-fail wiring gate.** `if self._dataLogger is not None and hasattr(self._dataLogger, 'getLatestReadings'): self._driveDetector.setReadingSnapshotSource(self._dataLogger)` is a Python-idiomatic guard but a silent-fail hazard -- if the attribute is missing, the entire downstream subsystem disarms with no log line. - **Pattern: synthetic regression gate must exercise the REAL wiring path, not a test double of it.** US-286 (Sprint 25 / Spool Story 3) shipped a regression gate that `setReadingSnapshotSource(_MutableSnapshotSource())` -- a hand-crafted mock that obviously exposes `getLatestReadings`. Standing rule: when you mock a seam, audit whether the mock side-steps a wiring constraint that production has to satisfy. Filtering by `WHERE drive_id = ?` would silently drop rows from the first N ticks of every drive -- exactly the rows that carry the cold-start ambient/baro snapshot. Use `WHERE timestamp_ms BETWEEN ? AND ?` to bypass the FK-NULL hole. This is a generic pattern: any backfill that runs against partially-corrupted data MUST query on observable timestamps, not denormalized FKs.

Standing rule: when a test asserts on a fetchone() result, either set `conn.row_factory = sqlite3.Row` (which makes Row tuple-comparable) or wrap the fetchone in `tuple()` explicitly. ### Learnings for future iterations:
- **Pattern discovered: trust the application's own emitted markers over systemd's late-shutdown markers under high log volume.** The intuition that "systemctl poweroff always emits 'Reached target Shutdown' / 'Powering off' / 'systemd-shutdown[1]:' to the journal" is wrong on rate-limited systems. systemd-journald drops messages under load; high-volume application logging plus an abrupt kernel halt drops late-shutdown messages BEFORE they flush to /var/log/journal. Standing rule for any future graceful-shutdown classification: prefer matching against application-emitted markers (which the codebase controls + can pair regression tests with) over framework markers (which can drop under load + drift across versions).
- **Pattern discovered: SSH for empirical pre-flight when the story acceptance literally says 'capture from production'.** US-308 acceptance step 1 was the canonical "go SSH the Pi and look at the actual journal" step. ### Learnings for future iterations:
- **Pattern discovered: one stable grep token per forensic surface.** B-071's whole value is "Drive 11+ produces a journalctl trail you can grep for known tokens." The discipline that makes that work: prefix every new instrumentation line with "FORENSIC <surface_name>" where <surface_name> is a fixed identifier (drive_check, drive_state_transition, drive_summary_writer_entry, ensureDriveSummary_entry, sync_push_table_entry, sync_push_table_advance, sync_push_drive_counter). The drill protocol then becomes `journalctl | grep "FORENSIC drive_state_transition"` -- a fixed token, never a regex over free text. Mirrors V0.27.2 US-307's `failed to close drain event` single-token pattern. The test asserts on the token substring, not the full message, so message wording can evolve without breaking the gate.
- **Pattern discovered: instrument the function ENTRY when the question is "did this run at all."** US-310/US-317 validation needs to know _ensureDriveSummary and captureDriveStart *ran* for a given drive_id even when the outcome was "deferred" or "UPSERT-no-change" -- not just when an INSERT happened. So the FORENSIC line goes at the very top of the function (after arg validation, before the first branch), not at the success path. This also means the test can mock everything below the log line: a MagicMock session + patched analytics is enough to assert the entry line fires, because the line fires before any of that machinery is touched.
- **Gotcha: spec method names can drift from code.** B-071's spec said "INFO at _checkDriveStart + _checkDriveEnd entry/exit" -- but `grep _checkDrive src/pi/obdii/drive/detector.py` returns nothing. The actual start/end-check logic lives in _processRpmValue (the STOPPED->STARTING->RUNNING and RUNNING->STOPPING->STOPPED transition decisions) + _transitionState (the state mutation) + _checkEcuSilenceDriveEnd (already WARNING-instrumented). This is a spec-phrasing-vs-code-naming gap, NOT phantom-path drift -- the FILE (detector.py) was in scope.filesToTouch, so instrumenting the real decision points inside it is in-bounds. The phantom-path stop-condition (BL-010 pattern) only fires when the *file* isn't in scope. Documented the resolution in the story's grounding so the PM sees it.
- **Gotcha: battery_health_log's PK is `drain_event_id`, not `id`.** When building a test SQLite fixture for the sync client's pushDelta('battery_health_log') path, the table MUST be created with `drain_event_id INTEGER PRIMARY KEY` -- sync_log.PK_COLUMN['battery_health_log'] = 'drain_event_id' (US-217), and _collectModifiedAt does `SELECT {pkColumn}, ... FROM battery_health_log` which errors with "no such column: drain_event_id" if you used `id`. The three SYNC_UPDATE_TABLES_PK tables are battery_health_log (drain_event_id), drive_summary (drive_id), dtc_log (id) -- only dtc_log uses `id`.
- **Gotcha: DriveState enum values are lowercase.** DriveState.RUNNING.value == "running". A test asserting `"RUNNING" in log_message` fails; assert `"running"`. The forensic line logs `state=%s` with self._driveState.value so the journal carries lowercase state names.

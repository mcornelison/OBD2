## 1. Bottom line

Restructure the backlog into a strict 4-tier hierarchy — Epic → Feature → Story → Task — with:

- structural data in `offices/pm/backlog.json` (SSOT machine-readable)
- prose contract in per-tier MD files (SSOT human-readable)
- PRD as a single-file workflow artifact (`prd.md` with YAML frontmatter for tooling-needed structured fields; multiple concurrent allowed)
- validation criteria as a required field on every Story (CIO 2026-05-23 directive #2)
- I-XXX / BL-XXX / TD-XXX intake artifacts folded into the hierarchy via a `type` field on Stories

Migrate ~10 active B-XXX items + ~30 V0.28+ candidates into the new shape; leave 35 completed B-XXX in legacy archive untouched.

## 2. Goals & non-goals

**Goals**
- Strict 4-tier hierarchy with no orphans (every Story has a Feature; every Feature has an Epic).
- Tooling-validatable contract (sprint_lint enforces required fields, type enum, JSON↔MD sync).
- Audit trail on every item (created/updated/completed timestamps).
- Multiple concurrent PRDs supported (Ralph executes Sprint N while PM grooms Sprint N+1).
- Validation criteria UPFRONT on every Story per CIO 2026-05-23 directive #2.
- Lowest-risk migration: re-tier active items only; legacy completed items stay untouched.

**Non-goals**
- Migrating 35 completed B-XXX items (out of scope; legacy archive shape stands).
- Replacing `sprint.json` (sprint.json schema gets minor additions; not rewritten).
- Tooling rewrite (existing scripts get schema-targeted edits; no greenfield).
- Defining the dev/main branching workflow (separate CIO directive #1; spec'd separately).

## 3. Architecture overview

```
[BACKLOG]                          [PRD]                              [SPRINT.JSON]
forever artifact                   per-sprint workflow (concurrent)   Ralph executable
SSOT: backlog.json + MD files      pm/prds/prd-<v>-sprint-<n>.md      offices/ralph/sprint.json
                                   (YAML frontmatter + markdown body)
                                                                       
4 tiers:                           Pulls groomed Stories               Generated by prd_to_sprint.py
  E-XXX  Epic                      from backlog (status: in-prd)       from prd.json + Story.md files
    F-XXX  Feature                 Refines goal/DoD/conditions/         (mechanical conversion).
      US-XXX  Story  ← unit         validation to "crystal-clear"      Adds sprint-level
        T-N  Task    (optional)     via Atlas + (optional) Argus       bigDefinitionOfDone
                                    reviews.                            (aggregates per-story
                                                                        validationCriteria).
```

**Tier semantics (standard agile):**
- **Epic** — multi-sprint capability, big-picture theme. Categorical (title + description).
- **Feature** — sprint-scope user value. Categorical (title + description + scope notes + risks).
- **Story** — INVEST-criteria day-or-two dev unit. **Substance lives here**: goal, DoD, conditional outcomes, validation criteria.
- **Task** — hours, dev-level breakdown. **Optional**; only added pre-sprint when grooming surfaces enough detail. Frozen at sprint dispatch (Ralph cannot add tasks mid-execution; TDD scratch lives in `progress.txt`).

**Storage:**

```
offices/pm/
  backlog.json                                  # SSOT structural tree (schemaVersion 2.0.0)
  backlog/
    E-001-uiux-polish.md                        # Epic descriptions
    E-002-data-pipeline-analytics.md
    ...
    F-103-splash.md                             # Feature descriptions (renamed from B-103)
    F-104-server-analytics-authority.md         # (renamed from B-104)
    ...
    US-359-boot-splash.md                       # Story substance (incl. inline Tasks)
    US-360-shutdown-splash.md
    ...
  prds/
    prd-V0.28.0-sprint-43.md                    # active PRD (Sprint 43, in-flight)
    prd-V0.28.0-sprint-44.md                    # next PRD (Sprint 44, in grooming)
  archive/
    completed-work-products/                    # graduated Epics/Features/Stories
    prds/                                       # archived PRDs alongside sprint archives
    intake-records/                             # original I-XXX/BL-XXX/TD-XXX after fold-in
```

**Tier status rollup:**
- Epic + Feature statuses are computed from children. `pm_status.py` is the **sole writer** of these fields: it recomputes on every invocation and writes the cached value back to `backlog.json`. Direct hand-editing of Epic/Feature `status` is prohibited (sprint_lint warns).
- Story + Task statuses are set MANUALLY with explicit lifecycles.

This caching-with-recompute design lets `grep`/`jq` queries against `backlog.json` return useful status without running tooling, while still preventing drift (recompute on next `pm_status` invocation overwrites stale cache).

**Graduation rule:** when an Epic/Feature/Story reaches `complete`, it's removed from `backlog.json` and its MD file moves to `offices/pm/archive/completed-work-products/`. Tasks travel inline with their parent Story.

### Field reference

| Field | Tier | Required | Notes |
|---|---|---|---|
| `id` | all | yes | `E-` / `F-` / `US-` / `T-` prefix per tier |
| `parent` | F/US/T | yes | Parent ID; Task's parent is implicit (lives in Story) |
| `title` | all | yes | Short, ≤70 chars |
| `description` | E/F | yes | Categorical context (1–3 sentences) |
| `type` | US | yes | Enum: `normal` / `issue` / `blocker` / `tech-debt` / `research` / `housekeeping` / `security` |
| `size` | US | yes | Enum: `XS` / `S` / `M` / `L` |
| `status` | all | yes | Per-tier status enum (see §5) |
| `goal` | US | yes | Connextra (`As <role>, I want <goal> so <benefit>`) OR Gherkin (`Given … when … then …`) |
| `definitionOfDone` | US | yes | Array of bulleted criteria |
| `conditionalOutcomes` | US | yes | Array of "if X then Y" rules |
| `validationCriteria` | US | yes | Array of `{action, outcome}` pairs (CIO 2026-05-23 directive #2) |
| `deps` | US | optional | Array of Story IDs (cross-Feature/Epic deps allowed) |
| `sourceRefs` | US | optional | E.g., `["B-103"]` for re-tier audit; `["I-040"]` for folded-in issues |
| `renamedFrom` | F | optional | Migration audit (B-XXX → F-XXX) |
| `tasks` | US | optional | Inline array of Task objects |
| `concreteOutput` | T | yes | What artifact gets produced |
| `addedAt` / `completedAt` | T | yes / when done | Task audit timestamps |
| `createdAt` / `updatedAt` | all | yes | ISO date |
| `completedAt` | all | when complete | Audit field set on graduation |

## 5. Status schema

| Tier | Statuses | Rollup |
|---|---|---|
| Epic | `pending` / `active` / `complete` | auto from features |
| Feature | `pending` / `groomed` / `in-sprint` / `active` / `complete` | auto from stories |
| Story | `pending` / `groomed` / `in-prd` / `sprint-ready` / `in-progress` / `blocked` / `passed` / `complete` | manual |
| Task | `open` / `done` | manual (with `addedAt` / `completedAt`) |

**Story lifecycle:**

```
pending      title + parent only; no contract yet
   ↓ (PM grooming: add goal/DoD/conditions/validation)
groomed      meets backlog contract; backlog-ready
   ↓ (PM pulls into upcoming PRD)
in-prd       pulled into a draft PRD for next sprint
   ↓ (Atlas review + optional Argus review; refinements applied to Story.md)
sprint-ready PRD-refined; in sprint.json (crystal-clear)
   ↓ (Ralph picks up)
in-progress  Ralph working
   ↓ (Ralph code-complete; tests pass; passes:true in sprint.json)
passed       acceptance + validation criteria met (sprint deploy gate)
   ↓ (sprint validated; merged to dev/main)
complete     graduates from backlog.json to archive/completed-work-products/
```

`blocked` can be entered from any in-flight state; story-MD includes a `## Blockers` section noting cause + clearance.

## 7. PRD lifecycle (per-sprint workflow artifact)

PRDs are **single-file markdown** (`.md` only) with YAML frontmatter carrying tooling-needed structured fields. **Multiple PRDs can be concurrent** (PM grooms Sprint N+1 while Ralph executes Sprint N).

**Why single-file, not dual:** the PRD is primarily a grooming/review document for humans (PM drafts; Atlas + Argus review; refinements logged narratively). Tooling (`prd_to_sprint.py`) only needs a handful of structured fields (selected story IDs, sprint number, version, status) — those fit cleanly in YAML frontmatter. The heavy structured data lives elsewhere: Story content in `Story.md`, structural data in `backlog.json`, Ralph-readable contract in `sprint.json` (generated at conversion).

### 7.1 PRD lifecycle states

```
[draft]            PM selects groomed Stories; opens with raw open Qs.
   ↓               Stories' status flips: backlog (groomed) → in-prd.
[atlas-review]     Atlas architecture review of selected stories + cross-cutting decisions.
   ↓
[argus-review]     QA review (only if PM/Atlas flags it needed).
   ↓
[sprint-ready]     Open Qs resolved; Story.md files updated with crystal-clear precision;
   ↓               sprint-level bigDefinitionOfDone aggregated from story validationCriteria.
[converted]        prd_to_sprint.py generates sprint.json; Stories' status flips: in-prd → sprint-ready.
   ↓               Ralph picks up. /sprint-deploy-pm at sprint end → statuses reach passed/complete.
[archived]         PRD moves to offices/pm/archive/prds/.
```

### 7.3 "Defined" (backlog) vs "crystal-clear" (PRD)

| Field | Backlog ("defined") | PRD ("crystal-clear") |
|---|---|---|
| `goal` | Connextra/Gherkin written; intent clear | No paraphrase ambiguity; one canonical reading |
| `definitionOfDone` | Bulleted criteria | Each bullet has a measurable check; no subjective "fast enough" |
| `conditionalOutcomes` | "if X then Y" rules | Each Y specifies system's exact response (state, timing, log line) |
| `validationCriteria` | (action, outcome) pairs | Each action has a script/command Argus can run; outcomes checkable post-condition |
| Atlas review notes | (none yet) | Per-task gate criteria pre-registered |
| Argus review notes | (none yet) | IRL drill methodology specified if needed |

### 7.4 Multiple concurrent PRDs

```
offices/pm/prds/
  prd-V0.28.0-sprint-43.md   <- Sprint 43 (status: converted; Ralph executing)
  prd-V0.28.0-sprint-44.md   <- Sprint 44 (status: draft; PM grooming next)
```

`pm_status.py` reports both: "Sprint 43 active (5/5 sprint-ready stories)" and "Sprint 44 in PRD grooming (3 draft stories, atlas-review queued)".

A Story can only be in ONE PRD at a time (status field enforces this). PM does not double-book Stories across PRDs.

## 8. Story `type` enum (folds I-XXX / BL-XXX / TD-XXX intake)

I-XXX (issues), BL-XXX (blockers), TD-XXX (tech-debt) prefixes retire going forward. New defects/blockers/debt are filed directly as Stories with a `type` field identifying origin.

| Value | Replaces | When to use |
|---|---|---|
| `normal` | (none — default) | Planned feature work |
| `issue` | I-XXX | Bug / defect |
| `blocker` | BL-XXX | Work that unblocks other work |
| `tech-debt` | TD-XXX | Known shortcut needing remediation |
| `research` | (new) | Research / investigation only; no shipped artifact |
| `housekeeping` | (new) | Operational hygiene (deploy scripts, lint, config) |
| `security` | (new) | Security-only work |

**Where typed Stories live in the hierarchy:**
- Bug clearly attaches to a Feature → file under that Feature (e.g., DriveDetector bug → F-107).
- No natural Feature home → file under standing **E-OPS Operational Hygiene** Epic with sub-Features (F-OPS-deploy, F-OPS-tooling, F-OPS-misc).

**Migration of existing intake records:**
- Open I-/BL-/TD- items get re-filed as Stories with `sourceRefs: [original ID]`.
- Original record moves to `offices/pm/archive/intake-records/` (audit preserved).
- `pm/issues/`, `pm/blockers/`, `pm/tech_debt/` folders retire (closed to new writes).
